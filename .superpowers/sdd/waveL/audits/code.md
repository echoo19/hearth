# Audit: code (example: ember-horde, port 5217)

## Findings

### CODE-1 · defect · high
- Element: Any typing inside the Code panel's script editor (CodeMirror surface, `.cm-content`) — the Space key specifically
- Observed: Pressing Space while the caret is inside the Code panel's editor inserts **nothing**. Reproduced with three independent methods, all consistent:
  1. Real keystrokes via `page.keyboard.type('hello world foo bar')` inside `.cm-content` → doc becomes `helloworldfoobar` (every space silently dropped); the identical call into a plain `<input>` elsewhere in the same app types correctly with spaces intact.
  2. A single isolated `keydown`+`keyup` of Space → zero characters inserted.
  3. `page.keyboard.insertText(' hello world')` (which synthesizes an `input` event directly, bypassing `keydown`) → inserts correctly, proving CodeMirror itself has no problem accepting a space character; only the discrete keydown path is broken.
  - Root cause: `apps/editor/src/components/SceneView.tsx`'s "hold Space to pan the canvas" handler (lines 476-495) is installed on `window` unconditionally (mounts whenever the Scene view exists, which is effectively always) and defines its **own**, incomplete local `isTyping` guard:
    ```js
    const isTyping = (t) => t instanceof HTMLElement && ['INPUT', 'TEXTAREA', 'SELECT'].includes(t.tagName);
    ```
    This does not check `isContentEditable`, unlike the shared `isTypingTarget()` in `apps/editor/src/keybinds.ts` (line 219-223), which correctly treats any `isContentEditable === true` element as a typing target. CodeMirror's `.cm-content` is a plain `<div contenteditable="true">`, not an INPUT/TEXTAREA/SELECT, so this local check misclassifies it as "not typing," and the handler calls `e.preventDefault()` on every Space keydown project-wide — including while a user is typing a script.
- Expected: Typing Space anywhere inside the Code panel's editor should insert a space character, same as every other printable key. This breaks writing virtually any line of real Lua/JS (any code with two identifiers, a binary operator with spacing, a string with words, etc.) — about as severe a defect as a code editor can have. Fix: use the shared `isTypingTarget()` (or check `isContentEditable`) in `SceneView.tsx`'s space-to-pan guard instead of its own duplicated, narrower tag list.

### CODE-2 · defect · high
- Element: External-edit detection (`.code-conflict-banner`) for a script modified **directly on disk**, bypassing the engine's command layer (e.g. a plain text editor, a raw shell `>>`/`sed`, a `git checkout`, or a coding agent that edits files with ordinary file-write tools rather than the Hearth CLI/MCP server)
- Observed: With `enemy-chase.lua` open and dirty in the Code panel, writing directly to `scripts/enemy-chase.lua` via `fs.writeFileSync` (simulating exactly the scenario in this audit's brief: "edit the same script file on disk with a shell command while it's open dirty") produces **no banner at all** — `.code-conflict-banner` count stays 0 even after waiting well past the journal watcher's poll interval. A subsequent Save from the dirty buffer would silently overwrite that external edit with no warning.
  - By contrast, the identical scenario driven through `packages/cli`'s `edit-script` command (`node .../hearth edit-script scripts/enemy-chase.lua ...`, which stamps its journal entry `source: 'cli'`) **is** detected correctly: the banner appears within ~3s with the expected copy ("This script changed outside the editor — from an agent or another process — while you had unsaved edits here."), and both resolution paths work — "Reload" adopts the CLI's content and clears dirty state; "Keep mine" dismisses the banner, keeps the local edit, and a subsequent Save (via the toolbar Save button) correctly overwrites the CLI's on-disk version with the local content.
  - Root cause: the whole mechanism (`apps/editor/server/journalWatcher.ts`) only watches `.hearth/log/commands.jsonl` — an append-only journal that ONLY the engine's own command layer writes to (the HTTP editor server, `packages/cli`, and `packages/mcp-server` each stamp a journal entry with `source: 'editor' | 'cli' | 'mcp'` on every command they execute). A raw filesystem write that never goes through that command layer produces no journal entry, so `fs.watch` on the log directory never fires, and the Code panel has no way to know the file under an open, dirty buffer changed.
- Expected: The feature's own stated invariant (`externalChange.ts`'s docstring: "an external agent's edit must never be silently discarded by a later Save from this panel") only actually holds for the subset of "external" edits that happen to go through the Hearth CLI or MCP server. Given the product's stated audience ("the 2D engine built for humans + coding agents") includes coding agents that may well use ordinary file-edit tools rather than the Hearth CLI, this is a real, currently-invisible clobber risk, not just an edge case. At minimum this gap should be a documented, deliberate limitation; ideally there'd be a lightweight mtime/hash check on Save (or a filesystem watch on `scripts/` itself) as a backstop for edits that never touch the journal.

### CODE-3 · friction · med
- Element: Pressing ⌘S (Cmd+S) while the Code panel is open but focus is NOT literally inside the CodeMirror text area (e.g. focus is on the "Format on save" checkbox, the script picker, or a conflict-banner button)
- Observed: The keypress falls through to the **global** `save` keybind (`apps/editor/src/keybinds.ts` lines 63-74), whose `run` does nothing but log `"Your changes are saved automatically — no need to save."` to the Console — it never calls the Code panel's own `save()`. Concretely: with a dirty script open, moving focus to the "Format on save" checkbox and pressing ⌘S leaves the tab's dirty dot showing and the file unsaved on disk, while the Console is told everything is saved. The Code panel's own `Mod-s` keymap (which does perform a real save) only intercepts the key while the caret is inside `.cm-content` specifically (confirmed: clicking the toolbar's own "Save" button in the same state does save correctly).
- Expected: The Code panel is the one place in this app where "everything auto-saves" is explicitly false (scripts require an explicit Save/Cmd+S, unlike scene/entity edits). The global keybind's reassurance message is actively misleading here — a user who just interacted with anything else in the Code panel and then hits the very natural ⌘S reflex gets told their (unsaved) changes are safe. At minimum the global binding should special-case an open dirty script buffer and route to the real save.

### CODE-4 · friction · med
- Element: Code panel's empty state hint text (`apps/editor/src/components/CodePanel.tsx` line 656-659) and the general lack of a "new script" affordance
- Observed: With zero scripts in the project, the panel shows: *"No scripts in this project yet — Create one with an agent, or from a Script component in the Inspector."* There is **no** "Script" component anywhere in the Inspector (grepped `Inspector.tsx` for any case-insensitive occurrence of "script" — zero hits beyond an unrelated doc-comment), and no "New script"/"Create script" button or context-menu entry in the Assets panel either. The **only** frontend caller of `createScript` is `CodePanel.tsx`'s own `saveAsNewScript()` (line 342), which is exclusively a recovery path for a script that's gone missing on disk mid-edit — not a general creation flow reachable from an empty project. The only real way to get a first script into a project today is via an AI agent (Timeline.tsx) or the CLI.
- Expected: Either the hint text should stop pointing at a non-existent Inspector affordance (half the sentence is simply false today), or a real "Script" component / "New script" action should exist in the Inspector/Assets panel to back up the claim. As written, a human user with no coding agent handy and a blank project has no discoverable path to create their first script.

### CODE-5 · unconfirmed (possible defect) · med
- Element: `ctx.` hover docs (`hoverTooltip` extension wired in `apps/editor/src/components/code/hoverDocs.ts` / `CodeEditor.tsx`)
- Observed: Hovering the mouse over a `ctx.scene.find`-style token (verified against the exact DOM span CodeMirror renders for that token, dead center, with fonts confirmed fully loaded) never produced a `.cm-tooltip` / `.cm-ctx-hover` element, across several methods: real OS-level mouse movement with a >1s dwell, and a directly-dispatched native `mousemove` event at the exact right coordinates on the exact right element. For comparison, `ctx.` **autocomplete** (a related CodeMirror tooltip mechanism, `.cm-tooltip-autocomplete`) rendered correctly and immediately in the same session/environment, and clicking to place the caret at the same coordinates worked correctly — both facts argue against a wholesale coordinate-mapping or tooltip-rendering problem in this test harness.
- Expected: Flagging with medium (not high) confidence specifically because mouse-hover-dwell automation in a headless Chromium+SwiftShare environment is a known-harder-to-trust interaction to synthesize than clicks/keyboard, and I could not rule out an environment-specific quirk in CodeMirror's internal `view.posAtCoords()` under software rendering. The implementation itself reads soundly on inspection (`ctxChainMatchAt`/`ctxDocAt` in `hoverDocs.ts` are pure and have their own unit tests per the file's doc comments). Recommend a quick manual sanity check in a real browser before treating this as confirmed.

## Verified working
- Opening scripts: the "Open a script…" dropdown in the Code panel toolbar lists every script under `scripts/` (label = filename without the `scripts/` prefix); selecting one opens (or activates, if already open) a tab and loads its content from disk.
- Console → Code jump (`openScriptAt`): a runtime script error during Play surfaces in the Console with a clickable `path:line` link (verified with a deliberately-injected `nil` index error in `player-move.lua`'s `onStart` — link read `scripts/player-move.lua:13`); clicking it switches the Code panel to front, activates the right tab, and flashes an ember-highlight on the exact line.
- Tabs: opening several scripts appends tabs in order; clicking a tab switches the active buffer; each tab shows its own independent dirty dot (•) based on that buffer's own saved-vs-live diff, not a global flag.
- Per-buffer undo isolation: typing distinct edits into two different open tabs, switching between them repeatedly, and undoing (Cmd+Z with the caret inside the editor) only undoes the ACTIVE buffer's own edit — the other tab's unrelated edit is untouched. A single `EditorView` correctly swaps cached `EditorState` per path (doc + selection + undo history) rather than remounting.
- Closing a dirty tab: clicking a tab's close (✕) button when it has unsaved changes shows the "Unsaved changes" confirm dialog ("Close X without saving? Your edits will be lost."); Cancel leaves the tab open and untouched; "Discard changes" closes it. Closing a clean tab has no confirmation step.
- Save (⌘S) semantics: pressing Cmd+S with the caret inside the editor on a dirty buffer saves it (dirty dot clears, content lands on disk byte-for-byte as typed when formatOnSave is off / format runs when on). Pressing Cmd+S again on an already-clean buffer is a true no-op (verified via `shouldSave` guard: no redundant write, Save button stays disabled).
- Format-on-save — Lua: badly-indented/unformatted Lua (`local script={}` etc., no spacing, no indentation) reformats correctly on Save to StyLua's house style (2-space indent, spaced operators, `end`/`function` blocks properly laid out) — verified via the actual on-disk file content before/after.
- Format-on-save — JS: unformatted JS (no semicolons, inconsistent indentation, `export default{...}`) reformats correctly via the "Format & save" button to Prettier's defaults (semicolons added, 2-space indent, expanded braces, trailing comma) — verified via on-disk content before/after. (No in-editor "new script" UI exists to create the `.js` file from scratch — see CODE-4 — so the file was placed on disk directly and picked up by the dropdown after any settings-refresh round-trip; format-on-save itself is fully verified once the file is open.)
- "Format & save" vs plain Save: "Format & save" always writes (and reformats) even on an untouched/clean buffer, matching its documented behavior that formatting can change on-disk bytes without a hand-edit.
- Lint/diagnostics: introducing a genuine Lua syntax error (an unterminated `if...then` block) surfaces a gutter error marker within the ~500ms debounce, with the exact engine-reported line/message (`checkScript`'s Lua parser correctly reports e.g. `[46:0] unexpected keyword 'if' near 'true'`); the dirty dot also correctly reflects the edit regardless of lint state. The lint round-trip is silent — it does not spam the Console panel.
- Autocomplete: typing `ctx.` inside either a Lua or JS buffer opens `.cm-tooltip-autocomplete` with the correct top-level `ctx` API members (verified list: animate, animator, audio, camera, clearSave, collisions, destroySelf, effects, entity, events, getComponent, input, isGrounded, load, log, math, params, particles, random, save, scene, …), sourced from the same `CTX_API` table the docs use.
- In-file search: Cmd+F while the editor has focus opens CodeMirror's own top-docked search panel (`.cm-search`); typing a query highlights matches (`.cm-searchMatch`) in the current buffer.
- Cross-script search (Shift+Cmd+F / the "Search" toolbar button): opens the `SearchAcross` bar; searching `ctx.vars.paused` correctly returns and groups matches by file ("13 matches in 3 scripts", grouped under `enemy-chase.lua` / `horde-director.lua` / `player-move.lua`); clicking a result row jumps the Code panel to that exact file/line with the ember-flash decoration, same as the Console-link path.
- Cross-script replace: expanding "Replace", entering a query + replacement, and clicking "Preview" runs a dry-run showing the exact per-file match counts before anything is written (verified: renaming `WAVE_SIZE` → `WAVE_SIZE_X` previewed "2 matches" in `horde-director.lua` only); clicking "Replace all" applies it for real, updates on-disk content correctly, and logs a clear summary ("Replaced 2 matches across 1 script.").
- Multi-file replace undoes as ONE step: after a real cross-script "Replace all" touching a file, clicking the toolbar's Undo button a single time fully reverts that file's content — confirming the engine's history records the whole bulk operation as one atomic entry, not one entry per touched file.
- External-edit conflict via the supported path (CLI/MCP, i.e. anything that goes through the command layer's journal): banner appears correctly on a dirty buffer within the journal watcher's poll window; both "Reload" (adopts the external content, clears dirty) and "Keep mine" (dismisses the banner, preserves local edits, and a subsequent real Save correctly overwrites the external file with the local version) work as designed. See CODE-2 for the raw-filesystem-edit gap in this same feature.
- Format on save checkbox: toggles cleanly (checked ⇄ unchecked) and persists to the project's `codeStyle.formatOnSave` setting; toggling it also triggers a full settings refresh that re-scans `scripts/` on disk (used during this audit to discover a script file placed directly in `scripts/`).

## Not covered
- CODE-5's hover-tooltip non-appearance could not be independently corroborated in a real (non-headless) browser within this audit's time budget — see the finding for exactly what was and wasn't ruled out.
- Project-switch behavior (Close project → open a different project, and what happens to open dirty script buffers) and the true zero-script empty-state screen were scripted for this audit but the run did not finish inside the audit window; both are real, testable scenarios worth a follow-up pass rather than a confirmed finding here. (Based on `store.ts`'s `closeProject()` being an in-place reset with no confirmation dialog of any kind — unlike the per-tab close confirm — the strong expectation is that unsaved script edits are silently discarded on project close with no warning, which would be a notable inconsistency with the per-tab UX; this needs direct confirmation before being filed as a finding.)
- Non-Mac keybind rendering/behavior (audited on macOS/Chrome only; Ctrl-based combos read but not exercised).
- Screen-reader/AT pass beyond the ARIA roles already visible in the tab strip (`role="tab"`/`role="tablist"`) and search results (`role="listbox"`/`role="option"`).
- A second concurrent browser/tab on the same project (true multi-client "another editor window" scenario for external-change detection) — only the CLI-sourced and raw-filesystem-sourced edit paths were tested.
- This environment shares its source tree with ~16 other parallel wave-L auditors' own dev servers; several test runs hit transient Vite/HMR breakage (a missing `@fontsource-variable/bricolage-grotesque` package, `vite-error-overlay` intercepting clicks) caused by concurrent, unrelated edits elsewhere in the repo — unrelated to the Code panel and not reported as findings, but it did shrink the time available for the deepest scenarios above.
