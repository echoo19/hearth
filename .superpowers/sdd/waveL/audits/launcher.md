# Audit: launcher  (example: ember-horde, port 5210)

## Findings

### LAUNCHER-1 · friction · med
- Element: `.launcher` page layout — `New project` card vs `Open a project` card (`Launcher.tsx`, `.launcher-columns`/`.launcher-card` in `styles.css`)
- Observed: With a populated Recent list (capped at 12) plus the Examples list (9 shipped examples), the "Open a project" card measures ~1310px tall against the "New project" card's ~432px — a 3x height mismatch. Total page content reaches ~1566px while the viewport audited was 950px, so the Examples section (and several Recent rows) sit below the fold with no independent scroll region: `.launcher-list` has no `max-height`/`overflow`, only the outer `.launcher` scrolls the whole page. A returning user with even a handful of recent projects never sees "Examples" without scrolling, and the two cards look badly out of balance since they're top-aligned (`align-items: start`) with nothing to equalize height.
- Expected: Cap/scroll the Recent and/or Examples lists independently (e.g. `max-height` + `overflow-y: auto` on `.launcher-list`, matching how many editors keep a launcher's history list internally scrollable), so the launcher reads as one balanced screen instead of a lopsided, scroll-required page.

### LAUNCHER-2 · friction · low
- Element: "Create project" / "Open" buttons while `busy` is true (`Launcher.tsx` lines ~143, ~162)
- Observed: Both buttons only go `disabled` (CSS: `opacity: 0.45`) while a create/open request is in flight; the label text never changes ("Create project"/"Open" stay static) and there's no spinner. On a fast local disk this is invisible, but on a slower filesystem (network share, cloud-synced folder) a user gets zero feedback that anything is happening beyond a slightly dimmer button.
- Expected: Swap in a "Creating…" / "Opening…" label (or a small inline spinner) during `busy`, consistent with giving real feedback for an action that isn't guaranteed to be instant.

### LAUNCHER-3 · polish · low
- Element: `.launcher-item` "Recent"/"Examples" rows — full path shown only via the native `title` attribute (`Launcher.tsx` lines 184, 210)
- Observed: Recent/Example paths are truncated with CSS ellipsis in `.item-path`, and the only way to see the full path is the browser/OS's native title tooltip (slow to appear, unstyled, and unavailable at all on touch). This matters more than usual here because duplicate project names are common (multiple "Ember Horde"/"Ember Trail"/"Glow Caves" entries were observed simultaneously in Recent during this audit, differing only by path — see LAUNCHER-4), so the path is often the *only* way to tell two same-named entries apart.
- Expected: A real (styled) tooltip, or reveal the full path on focus/hover in a small popover, rather than relying on the native title attribute for information that can be disambiguating.

### LAUNCHER-4 · friction · med (environment-scoped; verify before trusting in production use)
- Element: Recent-projects storage — `~/.hearth/recent-projects.json`, shared by every editor instance on the machine (`apps/editor/server/projectServer.ts` line 253-254: `options.recentsFile ?? path.join(os.homedir(), '.hearth', 'recent-projects.json')`)
- Observed: The recents file is a single global path, not scoped per window/instance/project-root. During this audit, the Recent list visibly interleaved entries from other concurrently-running editor sessions (other Wave-L auditors, in this case) — multiple duplicate-named entries ("Ember Horde" ×4, "Glow Caves" ×3, "Ember Trail" ×2) pointing at different scratch paths, and the list is capped at 12 total (`entries.slice(0, 12)` in `addRecent`), so a burst of activity in one window evicts another window's genuinely-recent projects. Read-modify-write in `addRecent` (read → filter → unshift → write, no lock) also has a plain TOCTOU race if two instances write around the same time — not proven to corrupt the file here, but the shared path is real and directly observable.
- Expected: At minimum this deserves a product decision — is the Recent list meant to be a single per-machine MRU across all open editor windows (arguably intentional, like most editors), or should concurrent instances not stomp each other's list? If intentional, it isn't a bug, but the duplicate-name confusion it produces (compounded by LAUNCHER-3's reliance on native tooltips for disambiguation) is worth a deliberate look.

## Verified working
- New-project name field: empty-name submit blocked with inline error "Give the project a name." (`.launcher-error`), autofocus on load confirmed.
- New-project Location field: pre-fills with `<home>/HearthProjects` placeholder once server meta loads; accepts custom absolute path; typed directory correctly becomes the parent of the created project's slugged subfolder.
- New-project Description field: optional; empty submits fine; text entered is correctly persisted into the created project's `hearth.json` (`description` round-tripped exactly).
- Template picker (`TemplatePicker.tsx`): renders as a true `role="radiogroup"` of 4 tiles (Blank, Platformer, Top-down, Arcade), Blank preselected by default and visually marked via `data-selected`.
- Template picker keyboard nav: focusing the group and pressing ArrowRight cycles blank → platformer → topdown → arcade → wraps back to blank; focus state and `checked` state and the visual `data-selected` card stay in sync at every step (verified programmatically across the full cycle).
- Template picker mouse selection: clicking the glyph icon, the label text, or anywhere else within a tile's `<label>` all correctly select that tile (native label-click-through works even though the glyph SVG visually overlaps the hidden radio input).
- Template picker focus ring: `.tpl-card:has(input:focus-visible)` gives a visible outline on keyboard focus, independent of the `data-selected` highlight.
- Create project (Blank): creates on disk, opens directly into the editor (Toolbar/Scene view mounts, Launcher unmounts), and the new project appears at the top of Recent on next visit to the launcher.
- Create project (Platformer template): scaffolds a populated scene (5 entities: camera, player, two platforms, autotiled ground), 1 script (`player-controller.lua`), 1 playtest, 2 assets; opens directly into the editor with the scene visibly rendered (player sprite + platforms + tiled ground on screen) — confirms "create from template, then confirm it opens" end-to-end.
- Duplicate create: submitting the same name+location twice surfaces the server's exact conflict message ("A Hearth project already exists at …") inline via `.launcher-error`, form stays on the launcher (no crash, no silent failure).
- Open-a-project, empty path: blocked with "Enter the path of a folder containing hearth.json."
- Open-a-project, nonexistent path: surfaces "Not a Hearth project: `<path>` has no hearth.json. Open the folder that contains hearth.json, or create a new project." — same message correctly covers "directory doesn't exist at all" and "directory exists but isn't a Hearth project" (server doesn't distinguish the two cases, which is reasonable — see note below, not filed as a defect since the message itself is already actionable).
- Open-a-project, valid path: opens correctly into the editor; the "Open" button and pressing Enter in the path field both submit identically.
- Recent list ordering: newly opened/created projects jump to the top (MRU); reopening a project already in the list moves it to the top rather than duplicating it.
- Recent list stale entries: entries whose folder no longer resolves to a project (`exists: false`, checked via `hearth.json` presence) render greyed out with "moved or deleted" in place of the path, and are `disabled` (not clickable) — confirmed via source (`disabled={busy || !r.exists}`) and visually in the initial-load screenshot (three such entries present before they aged out of the 12-item cap over the course of the audit).
- Recent list hover affordance: the trailing chevron (`.item-go`) is invisible at rest and fades in on hover/focus-visible — functions correctly, though this is the kind of hover-only affordance the audit brief calls out as worth flagging; here it's decorative only (doesn't gate a needed action) so not filed separately.
- Examples list: populated correctly from `packages/examples` (9 example projects listed with name + one-line description from each `hearth.json`); non-project subfolders (`fixtures`, `tests`, `generate.mjs`, `pixelart.mjs`) are correctly excluded (filtered by missing `hearth.json`). Clicking an example (Bounce Patrol) opens it directly into the editor.
- Tab order across the whole launcher: Name → Location → Description → Template radiogroup (lands on the single checked tile, per native radiogroup semantics) → Create project button → Open-path input → Open button → Recent items, in clean top-to-bottom, left-to-right document order with no traps.
- Branding/copy: header renders flame icon + "Hearth" + tagline "The 2D engine built for humans + coding agents" correctly; no layout shift or missing assets observed across all screenshots taken.
- `native` (Electron) feature-detection: correctly absent in this browser-only harness (`hearthNative()` returns null), so "Browse…" and "Open Folder…" buttons correctly don't render rather than rendering broken.

## Not covered
- Electron-only affordances (`Browse…` directory picker, `Open Folder…` native picker, `revealInFolder`) — `window.hearthNative` is only injected by the Electron preload script; this harness drives a plain Chromium tab against the Vite dev server, so these paths are structurally unreachable here. Not exercised.
- One non-reproducible anomaly: in the very first end-to-end create-project run of this session (immediately after exercising keyboard nav + forced glyph clicks on the template picker), the app appeared to bounce back to a freshly-mounted, empty-state Launcher (Recent/Examples both showing their "nothing yet" empty copy, form fields reset to placeholders) instead of transitioning into the editor — even though the project *was* successfully created on disk (confirmed via a follow-up duplicate-name conflict check, and via the server's recents file). Re-running the identical interaction sequence twice more (including once via a byte-for-byte rerun of the same script against a fresh target directory) reproduced the normal, correct behavior both times (toolbar mounted immediately, no reset). Given 1-in-3 and no repro after targeted retries, logged here as an observation rather than a scored finding — worth a second look if it recurs, since a full silent bounce-to-launcher-with-empty-state after a successful create would be a real (if rare) defect.
- Concurrent-write corruption of `~/.hearth/recent-projects.json` under true simultaneous writes (LAUNCHER-4) — the shared-file behavior and duplicate/eviction symptoms were directly observed, but no actual data loss or corrupted JSON was caught in the act; flagged as a design concern rather than a proven corruption bug.
- Screen-reader output (VoiceOver/NVDA) was not exercised — only DOM semantics (`role="radiogroup"`, native radio/label pairing, disabled state) were verified via evaluate() and the accessibility-relevant CSS (focus-visible outline, visually-hidden-not-display-none inputs).
- Resizing the launcher to narrow/mobile widths beyond the one documented breakpoint (`@media (max-width: 380px)` drops the template picker to a 2-up grid) was not visually verified — only read from source.
