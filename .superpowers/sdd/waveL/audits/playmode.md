# Audit: playmode  (example: drift-cellar then ember-horde, port 5219)

## Findings

### PLAYMODE-1 · defect · high
- Element: keyboard movement while Playing (WASD keys — `moveX`/`moveY` virtual axes, bound in `hearth.json` → `inputMappings.axes.moveX/moveY.negativeCodes/positiveCodes` to `KeyA/KeyD`/`KeyW/KeyS`)
- Observed: After pressing Play, holding `KeyD` (or any letter-only axis code) for a full second — even from a fully settled, zero-velocity state — produces **zero** change in the player's live velocity or position. Confirmed two ways:
  1. Live panel: `Velocity x: 0.00, y: 0.00` and `World pos` unchanged across a clean 1s `KeyD` hold, repeated with settle time in between (ruled out warm-up race and residual-momentum artifacts from a prior key).
  2. Direct instrumentation: wrapped every `window.addEventListener('keydown'/'keyup', …)` call before the app loaded and logged `event.defaultPrevented` as each of the 13 real listeners fired for a `KeyD` press — **none** ever set `defaultPrevented = true`, meaning the runtime's own `attachKeyboard()` handler (`packages/runtime/src/pixi/index.ts:545-550`) never recognizes `KeyD` as mapped and never calls `handleKeyDown`.
  - Root cause: `InputState`'s `codeToActions` map (`packages/runtime/src/input.ts:111-140`) is built only from `mappings.actions`, `gamepadButtons`, and `gamepadAxes` — it never includes `mappings.axes[*].negativeCodes/positiveCodes`. `attachKeyboard()`'s keydown handler only calls `handleKeyDown(code)` when `isMappedCode(code)` is true (`pixi/index.ts:546-549`), so any axis-only code is silently dropped and never reaches `downCodes`, which `axisValue()` (`input.ts:272-289`) reads for the keyboard fallback.
  - In Drift Cellar, **ArrowKeys happen to work** only because they are *also* bound to the `ui-up/down/left/right` named actions (so they legitimately pass `isMappedCode`) — this is a coincidence of this project's input map, not a fix. WASD (`KeyA/KeyD/KeyW/KeyS`) is bound to *only* the axis, so it is completely dead in every direction, in the editor's Game preview and (since this is `packages/runtime`, not editor-only code) presumably in exported/built games too.
- Expected: The in-game hint text literally reads "Stick/WASD drifts — Space dashes — Esc pauses." WASD should move the player exactly like the arrow keys do. This is the primary documented control scheme for the example the audit brief called out for "good UI coverage," and it does not work at all via play-testing in the browser.

### PLAYMODE-2 · defect · med
- Element: switching the active center tab away from "Game" while a run is in progress (e.g. clicking the "Code" tab, which by default lives in the same tab stack as Scene/Game/Code)
- Observed: The moment the Game panel loses visibility, `GamePanelHost` (`apps/editor/src/workspace/Workspace.tsx:98-106`) calls `setPlaying(false)` — this is a full **Stop**, not a soft pause: it resets `pendingRestart`, clears `runtimeErrors`, and increments `runNonce` so the next Play fully remounts the runtime from scratch. Reproduced cleanly twice: toolbar read "Stop" before switching tabs, "Play" immediately after, and the Live panel reverted to its empty ("Press Play…") state.
- Expected: The code comment says "pause the preview when its tab is hidden" — but the implementation is a hard Stop, losing all play-session state (position, timers, HUD counters, spawned entities), not merely freezing rendering the way the toolbar's own debug Pause does. This directly undercuts the "hot-reload during play" workflow this audit was asked to exercise: the obvious way to edit a script while watching Game live — open the Code tab — kills the run first, because Code/Scene/Game are tabs in one stack by default (`CENTER_PANELS`/`buildDefaultLayout` in `Workspace.tsx`). The only way to hot-reload with Game still visible is to manually drag the Code tab out into its own split panel, which is not the default or a discoverable layout for this feature. At minimum the comment/behavior mismatch (says "pause," does "stop") should be resolved, and ideally switching to Code shouldn't kill an in-progress run at all.

### PLAYMODE-3 · friction · low
- Element: toolbar "Pause" (debug pause) with no in-canvas indicator
- Observed: Clicking Pause freezes the frame counter deterministically (verified — see Verified working), but there is no visual cue anywhere inside the Game canvas itself that the game is frozen; only the toolbar button relabels itself "Resume."
- Expected: A frozen game with no on-canvas signal (no dimming, no "PAUSED" overlay, no border treatment) can read as a hang/crash rather than an intentional debug freeze, especially since the sprites keep rendering their exact last-frame pose. A subtle overlay or border tint while debug-paused would remove the ambiguity.

### PLAYMODE-4 · friction · med (performance)
- Element: Game preview panel frame pacing under load (ember-horde, 300 concurrent enemies)
- Observed: Sampling the Live panel's frame counter every 1.5s while the horde director spawned its capped 300 enemies (`ENEMY_CAP = 300`, `packages/examples/ember-horde/scripts/horde-director.lua`) showed the effective simulation rate falling steadily as entity count grew:
  - ~106 entities: ~53 fps
  - ~196 entities: ~49 fps
  - ~256 entities: ~33 fps
  - ~306–316 entities (full horde): sustained **~18 fps**, roughly a 3x drop from the low-entity-count baseline, holding steady (not still declining) once the cap was reached.
  - This ran under headless Chrome + SwiftShader (software GL), so the *absolute* fps numbers are not representative of a real GPU; the environment was constant across all samples, though, so the ~3x relative degradation as entity count triples is a fair signal that per-frame cost scales worse than linearly somewhere in the sync/update path, worth a profiling pass (candidate: `PixiSceneView`'s per-frame entity sync in `packages/runtime/src/pixi/index.ts`).
- Expected: A demanding "does this feel tight in 1.0" bar would want the preview panel to hold close to target fps (60, per this project's `buildSettings.targetFps`) at its own documented enemy cap, or at least degrade gracefully/linearly rather than to roughly a third of the mid-load rate.

## Verified working
- Play/Stop toolbar button toggles label, icon, and color correctly; Play remounts the runtime fresh each time (`runNonce` bump).
- Live panel empty state when stopped: exact copy "Press Play to inspect the running game." plus the descriptive hint about transform/velocity/timers/tweens/events, matching `LivePanel.tsx`.
- Live panel populates correctly once Playing: entity dropdown, frame counter, entity count, Name/ID/Tags/Enabled/World pos/Velocity rows, Timers/Tweens/Recent events cards all render with the right fields for the selected entity.
- Live panel auto-seeds its entity selection from the current scene selection (Player) when a run starts, and keeps it live across frames.
- Live panel polling is real-time and ~10Hz: frame counter reliably advanced between samples taken 100–400ms apart across every test.
- Debug Pause (toolbar) freezes the frame counter exactly — confirmed motionless across a 500ms hold at the same frame number while the game stayed mounted and responsive.
- Step (toolbar, disabled unless Paused) advances **exactly one** frame per click — confirmed frame N → N+1 on click, then frame stays at N+1 for another 500ms with no further drift; a second Step click advances exactly one more.
- Resume continues frame advancement smoothly and immediately after a Pause/Step sequence.
- Debug overlay toggle (toolbar "Debug" button): turns on/off collider outlines (walls, player, gems) cleanly in the Game view; button shows pressed/active state; toggling off cleanly removes all overlay draws with no leftover artifacts.
- Hot-reload on a **successful** script save: editing `player-drift.lua`'s speed constant (240→480) and saving produced a clean `Hot-reloaded scripts/player-drift.lua (1 entity)` Console line; the player's live position and the run's frame counter continued uninterrupted (no restart, no reset to spawn); a follow-up movement test confirmed the new speed value was actually driving the physics (post-reload displacement over a fixed hold was markedly larger than the pre-reload baseline at the old speed).
- Hot-reload on a **failing** script save (accidental Lua syntax error from my own edit): Console clearly surfaced both `Save script: Could not format … error parsing: … unexpected token … expected 'end'` and a follow-up `Hot-reload failed: scripts/player-drift.lua:40 — … 'end' expected (to close 'if' at line 38) near <eof>`, with a clickable script:line reference; the game did not crash or freeze — the frame counter kept advancing and the player kept responding to input, confirming the previously-good script kept running.
- Live property patching via the Inspector: editing Player's `Transform.Position.x` to 500 while Playing applied immediately to the running entity (confirmed in the Live panel's World pos) with **no** restart badge — a clean live patch, no restart needed.
- Structural change via the Inspector while Playing (adding a new component, e.g. Camera, to Player): correctly surfaced the "Scene changed — Restart" badge (exact copy) instead of attempting to live-patch.
- Clicking the restart badge: clears the badge and restarts the run from scratch (frame counter dropped back down, confirming a genuine remount) rather than resuming stale state.
- Stop: Live panel cleanly reverts to its empty state; a fresh Play afterward starts from a clean, correct baseline (frame low, velocity zero, no leaked timers/tweens/events from the previous run, Tags/Enabled/authored fields intact).
- Escape (mapped to the `pause` action, which — unlike WASD — also works correctly since it's action-bound) opens Drift Cellar's in-game pause menu: "Paused" title, Resume button auto-focused with its highlighted color, Music Volume slider and Screen Shake toggle all rendered and legible.
- Mouse click directly on the Screen Shake checkbox toggles it (visual color swap from checked/orange to unchecked/gray), confirmed via screenshot diff.
- Mouse drag on the Music Volume slider handle moves it and updates the filled-track visualization correctly (dragged fully to the low end, track fill emptied out).
- Mouse click on Resume closes the pause menu and un-pauses gameplay — confirmed the player resumed responding to movement input immediately after.
- ArrowKey-driven movement (as opposed to WASD) works correctly end-to-end: smooth drift/momentum-style easing toward target velocity, matching `player-drift.lua`'s lerp logic.
- ember-horde's 300-enemy horde spawns to its documented cap (HUD correctly reads "Enemies: 300/300"), and the Game preview keeps rendering correctly under load — no visual corruption, no crash, HUD text (Time/Enemies/HP) stays legible and updates live throughout.

## Not covered
- Gamepad input: not simulatable via headless Playwright/CDP in this environment; `isMappedCode`/`pollGamepads` gamepad-button and gamepad-axis paths were read in source but not exercised live.
- Keyboard-driven slider/toggle adjustment inside the pause menu (`ui-left`/`ui-right` → `ctx.ui.adjust`) — only the mouse-drag/click paths were exercised live; the keyboard-nudge path was read in `menu-controller.lua` but not driven end-to-end.
- Camera zoom-punch/flash/shake visual effects (dash, wall bump) — inferred from code and indirectly exercised via movement tests, not isolated and screenshot-verified frame-by-frame.
- Scene-switch mid-play (Director's fade-to-black → "Vault" scene load on collecting all 3 gems) — not driven to completion (would have required navigating the player into all three gems under the WASD/arrow input constraints discovered above).
- Absolute frame-rate numbers for the ember-horde load test are only meaningful in relative terms — this ran under headless Chrome with SwiftShader software GL rendering, not a real GPU, so the ~18fps floor should not be read as what a real machine would see; the ~3x relative degradation trend as entity count scales is the load-bearing part of PLAYMODE-4.
- A second concurrent auditor's active source edits (visible via this dev server's Vite HMR log, e.g. mid-session invalidations of `Workspace.tsx`, `Toolbar.tsx`, `SceneMenu.tsx`, and a brief window where `packages/core/dist` didn't export `AGENT_SKILL_CONTENT`, breaking `vite.config.ts` entirely for ~30s) caused several transient, unreproducible page crashes and one full project-load failure during this session. Those were treated as environment noise, not filed as findings, and every finding above was independently reproduced on a clean run with no console errors of that kind present.
