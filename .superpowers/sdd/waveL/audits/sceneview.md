# Audit: sceneview  (example: glow-caves, then ember-horde, port 5213)

## Findings

### SCENEVIEW-1 · defect · high
- Element: any `UIElement` entity rendered/selected/dragged in the Scene view (e.g. ember-horde's "Timer HUD", "HP HUD", "Horde HUD", "Hint", "Pause Menu" and its children "Menu Title"/"Resume"/"Shake Label"/"Screen Shake").
- Observed: `SceneView.tsx` positions every entity purely from `Transform.position` walked up the `parentId` chain (`worldPos`/`localPos`); it never reads `UIElement.anchor`/`UIElement.offset`. Since every UI-anchored entity in ember-horde's scene has `Transform.position: {x:0, y:0}` (by design — the real screen position comes entirely from anchor+offset, see `packages/runtime/src/ui.ts`'s `uiScreenPosition`, which never looks at `Transform.position` either), all of them collapse to the exact same world point and render as illegible overlapping text at the scene's world origin instead of at their four distinct screen anchors (top-left, top-right, top-center, bottom-center, center). Repro: open ember-horde's Arena scene in the Scene view — "Paused", "Resume", "Screen shake" and the "Stick/WASD…" hint text are all stacked on top of each other near world (0,0), nowhere near where they actually appear on screen during Play. The entities are still selectable and draggable there (resize handles appear, `moveEntity` fires), but editing them in the Scene view is functionally meaningless: the runtime computes their real position from anchor+offset alone, so a Scene-view drag never affects where the label actually shows up in the game.
- Expected: UI-anchored entities should render (and be selectable/draggable, or explicitly be excluded from free drag) at their resolved anchor+offset screen position, matching `uiScreenPosition`'s math, so the Scene view is a truthful preview of the HUD layout instead of a pile of overlapping text at the origin.

### SCENEVIEW-2 · friction · low
- Element: toolbar "Debug" button, visible/enabled while the Scene tab is active.
- Observed: toggling Debug has zero visual effect on the Scene view — verified by screenshotting the same selection/gizmo state with Debug on and off (pixel-identical). `SceneView.tsx` never reads `store.debugDraw`; that flag only affects the Game tab's runtime overlay (`packages/editor` Game preview). The Scene view's own gizmos (collider box/circle, Light2D radius, particle direction cone, camera viewport box) always render unconditionally and can't be hidden.
- Expected: either gray out / relabel the button while on the Scene tab (it does nothing there), or wire it to also toggle the Scene view's own gizmo layer so "Debug" means one consistent thing across tabs.

### SCENEVIEW-3 · friction · low
- Element: entity move-drag and transform-handle resize-drag; the 32px background grid is always drawn.
- Observed: nothing snaps to the grid. Move-drag rounds the final position to the nearest integer pixel on commit (`Math.round`) but never to a grid multiple; resize handles round to the nearest integer px (or 0.01 for `Transform.scale`) but likewise never snap. The only snapping anywhere in the gizmo system is the rotate handle's optional 15° shift-snap.
- Expected: given the grid is drawn as a first-class visual aid, a modifier-triggered (or default) snap-to-grid for move/resize would match user expectation set by the grid's presence; today the grid is purely decorative.

### SCENEVIEW-4 · defect (unverified, source-substantiated) · med
- Element: entity drag (`onEntityPointerDown`/`onPointerMove`/`onPointerUp`) and tilemap paint (`startPaint`/paint pointer handlers) while `playing` is true.
- Observed: the transform-handle gizmo is explicitly hidden while playing (`showHandles = !playing && ...`), but the entity-drag-to-move path and the tilemap-paint path have no `playing` check at all — a drag or paint stroke while `playing === true` would still call `moveEntity` / `paintTiles` / `fillTilemapRect` against the live scene. In the shipped single-pane tabbed workspace this looks unreachable in practice: `Workspace.tsx`'s `GamePanelHost` stops Play the instant its panel is hidden (`onDidVisibilityChange` → `setPlaying(false)`), and switching to the Scene tab hides the Game panel, so Play always turns itself off the moment you leave the Game tab. I confirmed this transition live (Play → switch to Scene tab → toolbar reverts to "Play", not "Stop").
- Not fully verified: dockview supports splitting a tab into its own panel group, which would keep both Scene and Game simultaneously visible (and thus keep Play running) without hiding either. If that's a supported workspace layout, nothing stops a drag or paint stroke in the still-visible Scene panel from mutating the scene while the game is actually running underneath. I attempted to reproduce this by dragging the Game tab into a split group, but the drag gesture left the automated browser session unresponsive and had to be restarted before I could complete the check (see Not covered). Given the asymmetry with `showHandles`'s explicit playing-guard, this is flagged as a likely real gap rather than confirmed.

### SCENEVIEW-5 · polish · low
- Element: polygon Collider / LineRenderer vertex drag (`vertexDragRef` live-drag path).
- Observed: while dragging a vertex, only the orange edit-mode outline overlay (`draftPoints`) updates live; the actual `renderLineRenderer`/collider-fill visual underneath keeps showing the pre-drag shape until the drag commits on pointer-up (since it reads the real, uncommitted `lr.points`/`Collider.points`). For a stroke width thick enough to be visible (e.g. glow-caves' "Cave Wall Left", width 6), this produces two visibly different overlapping line renderings during the drag — the stale thick line and the live thin orange guide — which briefly reads like a rendering glitch to someone not expecting it.
- Expected: no strong expectation either way, but a single, consistent live shape during the drag (e.g. suppressing the stale render or updating it from the same draft points) would remove the double-image moment.

## Verified working
- Click-select an entity via the Hierarchy panel and via clicking it directly in the canvas — Inspector populates, orange selection outline + resize/rotate handles appear, Hierarchy row highlights, HUD shows world position.
- Click on empty canvas background deselects (Inspector reverts to "Nothing selected", outline/handles disappear).
- Escape deselects the current selection (confirmed via DOM `aria-selected`/Inspector state, not just visually).
- Escape does not fire while a typing target (Inspector text input) is focused, nor while `editingPoints`/`paintMode` own their own Escape.
- Drag-to-move an entity: live ghost during drag, commits one rounded-integer `moveEntity` on release; one Undo fully reverts, Redo reapplies.
- Resize via an edge handle (`e`) edits only the corresponding axis (width, not height), center-anchored (position unchanged); one Undo reverts.
- Resize via a corner handle (`se`) edits both width and height together, batched into a single `setProperties` command — confirmed one Undo reverts both dimensions at once.
- Shift+corner-drag resize is aspect-locked: dragging a 28×32 sprite mostly horizontally by shift-drag grew it to 50×57, preserving the ~0.875 aspect ratio (within integer-rounding).
- Rotate handle: dragging to world-east from the top-stem handle set `Transform.rotation` to exactly 90, matching the documented clockwise/SVG-y-down convention; commits as one Undo step.
- `resolveHandleTarget` priority confirmed live: an entity with both `SpriteRenderer` and a circle `Collider` (ember-horde's "Enemy") shows the sprite-size square handle box, not a circle-adorned box — SpriteRenderer correctly wins over the collider per the documented priority order.
- Camera pan: space+drag and middle-mouse-drag both pan the view by exactly the pointer delta.
- Wheel zoom in/out around the cursor point; keyboard `=`/`-` zoom by the documented 1.25× step around the viewport center; `0` fits the view to the build viewport.
- Keyboard zoom shortcuts correctly do nothing while a text input (Inspector "Name" field) has focus (`isTypingTarget` guard).
- Polygon/LineRenderer point editor ("Edit points"): drag an existing vertex to a new position (committed, rounded to 2 decimals); click an edge midpoint to insert + commit a new vertex exactly at the midpoint; alt-click (or right-click, same branch) a vertex deletes it, refusing below the component's minimum point count (guarded by `removeVertex`'s `min` parameter — code-verified for both the 3-point polygon-Collider case and the 2-point LineRenderer case, live-tested for the LineRenderer path since no polygon Collider entity exists in either example project).
- "Done"/Escape exits point-editing mode cleanly; switching entity selection while editing exits it via the `canEditPoints` effect.
- Tilemap paint tool (glow-caves' "Cave Rocks"): toggling "Paint tiles" shows the palette (eraser + one swatch per tile char) and updates the cursor/hint text; single-cell click paints exactly the clicked cell and autotile-resolves neighbors on commit; shift-drag previews a translucent rect live and commits a `fillTilemapRect` that blends into the existing autotile blob; eraser click clears a single painted cell without touching neighboring cells; Escape or "Done painting" exits the mode.
- Tilemap "Resize…" dialog: growing (10×8 → 12×10) pads new cells as empty on the right/bottom; shrinking (12×10 → 6×5) crops from the right/bottom, matching the dialog's own copy.
- Particle emitter live in-scene preview (glow-caves' "Drip"): selecting an emitting entity starts a live particle stream (falling/fading droplets matching `startColor`→`endColor`, `startSize`→`endSize` interpolation); dragging the entity's origin translates the whole stream smoothly with the drag ghost, without resetting/restarting the particles.
- The "Particles" toggle and its localStorage-backed preference are read on mount (`hearth.scene.particlesPreview`).
- Selected-entity outline correctly tracks position/rotation/scale for ordinary Transform-positioned entities (sprites, tilemaps, line renderers, box/circle colliders) in both example projects.
- Play/Stop: Play switches to the Game tab and starts the runtime (torch light + ambient darkening rendered correctly in ember-horde/glow-caves); Stop returns control; switching back to the Scene tab while "playing" auto-stops Play (`GamePanelHost`'s visibility-based stop), confirmed live.
- "Close project" (toolbar) correctly returns to the launcher and the just-closed project appears at the top of "Recent"; reopening a second example project (ember-horde) via the "Open a project" path field works cleanly, including its own instance of the documented "all panels are closed" dockview bug and the described View-menu toggle-off/toggle-on fix.
- The known dockview "all panels are closed" bug reproduced on every fresh project open in this session (both examples, multiple times) and was fixed each time by toggling the affected panel off then on from the View menu, exactly as documented.

## Not covered
- True rectangle/marquee multi-select: does not exist in the code at all (confirmed by full read of `SceneView.tsx` — only single-entity click-select is implemented, no drag-rectangle selection path), so there was nothing to exercise.
- A `Collider.shape === 'circle'` entity *without* a `SpriteRenderer* (which would surface the dashed-circle resize target visual) does not exist in either glow-caves or ember-horde — every circle-collider entity in both example projects also has a sprite, so `resolveHandleTarget` always resolves to `sprite-size` first (verified correct per its documented priority order) and the circle-specific gizmo rendering path was only confirmed by code reading, not visually.
- A `Collider.shape === 'polygon'` entity does not exist in either example project; the shared point-editor math (`polygonEditing.ts`) was fully exercised instead through the LineRenderer vertex editor, which uses the identical transform/edit code path (see `pointsSourceFor`), just not the polygon-Collider component swap itself.
- SCENEVIEW-4 (playing-state guard gap for entity drag / tilemap paint) could not be confirmed against an actual simultaneous Scene+Game split-panel layout: a dockview tab-drag-to-split gesture left the automated browser session unresponsive and had to be restarted, and time did not allow a second attempt with a different drag technique.
- One tilemap-resize (shrink) attempt was interrupted mid-test by an unrelated Vite dev-server error overlay (`[plugin:vite:css] ENOENT ... @fontsource-variable/bricolage-grotesque/index.css`) that blanked the whole app behind a full-screen error dialog; this is an environment/dependency issue outside SceneView's own code (not scored here), and the resize was successfully re-verified afterward once the overlay was dismissed.
- Right-click (button 2) as the alternate vertex-delete trigger was not separately exercised (same code branch as the alt-click path that was tested: `e.button === 2 || (e.button === 0 && e.altKey)`).
