# Audit: exportdialog  (example: mini-platformer, port 5225)

## Findings

### EXPORTDIALOG-1 · defect · medium
- Element: Native `<dialog>` root (`ui.tsx` `Modal`: `<dialog className="modal" ref={ref} onCancel={onClose} onClose={onClose}>`), exercised while a Desktop export is running.
- Observed: While a desktop export is actively running, both the segmented control and the modal's own "Cancel" button are correctly disabled (confirmed: `Cancel button disabled attr while running: true`; clicking it is a no-op). However, pressing **Escape** fires the native `<dialog>` `cancel` event, which is wired unconditionally to `onClose` with no running-job guard. Repro: start a darwin-arm64 desktop export → wait for the row to reach "Packaging"/"Building" → press Escape → the dialog closes immediately (`Open dialog count AFTER Escape while a desktop export is RUNNING: 0`), even though the visible Cancel button one row below was disabled specifically to prevent this. The job itself is unharmed (it's tracked in a module-level store independent of the dialog's mount state) — reopening shows it still progressing correctly with the tab-lock intact — so no data is lost, but the interaction is inconsistent: mouse users are blocked from dismissing mid-export, keyboard (Escape) users are not.
- Expected: Either the `onCancel` handler should also check `jobRunning` (mirroring the Cancel button's `disabled` guard) and swallow the Escape while a desktop export is running, or the running-state affordance should make clear that Escape still works (currently it implies the opposite). This is in the shared `Modal` component (`ui.tsx:299`), but only ExportDialog has a "some button in me is deliberately disabled to block dismissal while busy" contract, so it's the surface where the gap actually bites.

### EXPORTDIALOG-2 · polish · low
- Element: `Modal`'s `<div className="modal-title">{title}</div>` (`ui.tsx:302`), used for the "Export" heading.
- Observed: The dialog's visible title is a plain `div`, not associated with the `<dialog>` via `aria-labelledby` (and the `<dialog>` itself has no `aria-label`). Native `showModal()` gives the dialog an implicit modal role, but with no accessible name wired up, assistive tech gets an unlabeled dialog rather than one announced as "Export".
- Expected: `<dialog aria-labelledby="export-dialog-title">` + `id="export-dialog-title"` on the title div (or an `aria-label` echoing `title`) would give the dialog a real accessible name. Shared-component root cause (`ui.tsx`), surfaced here because Export is the dialog most worth naming correctly for a screen-reader user picking the right modal.

## Verified working

**Opening / chrome**
- Toolbar "Export" button opens the dialog; focus visibly returns to the toolbar button after the dialog closes (Escape), consistent editor-wide focus-return behavior.
- `role="tablist"` (`ModeSwitch`) / `role="tab"` (Web, Desktop) / `role="tabpanel"` (`export-pane-web` / `export-pane-desktop`) wired correctly: tab `aria-controls` matches the panel `id`, panel `aria-labelledby` matches the tab `id`, `aria-selected` flips correctly, roving `tabIndex` (0 on active tab, -1 on inactive) confirmed in the DOM.
- Segmented control keyboard nav: focus on "Web" tab + ArrowRight moves selection to "Desktop"; ArrowLeft moves back — confirmed via `aria-selected`/active-class state, and confirmed the handler respects `disabled` (no-ops while a desktop job is running).
- Both tabs (and the whole segmented control) become `disabled` while a desktop job is running, locking the user on the Desktop pane — confirmed via `disabled` attribute and visually (dimmed control).

**Web pane**
- Format radios ("Web build folder" / "Single HTML file") are mutually exclusive, selection reflected in `.radio-row:has(input:checked)` accent styling.
- Output-directory field accepts free text; Enter key in the field triggers export (same as clicking the primary button).
- "Zip for itch.io" checkbox toggles independently of the format radios.
- **Real end-to-end web export (folder format + zip checked)**: POST succeeded, wrote 14 files to `export/web` (verified on disk: `index.html`, `hearth-player.js`, `project.bundle.json`, `assets/`), plus `export/mini_platformer-web.zip` next to it. Unzipped and confirmed `index.html` sits at the **zip root** alongside `assets/`, `hearth-player.js`, `project.bundle.json` — not nested inside a subfolder. Slugification confirmed (`mini-platformer` project → `mini_platformer-web.zip`, hyphens→underscores).
- **Real single-HTML-file export**: format = "Single HTML file", ran export, result correctly reports "Exported 1 file" and lists exactly one path (`export/single`).
- Result panel: renders file count, absolute out-dir path, absolute zip path (when zipped) with `title=` tooltip carrying the full path for truncation cases, and a working "Copy path" button next to the zip line.
- Copy-path button: click copies the exact absolute zip path to the clipboard (verified via `navigator.clipboard.readText()`), swaps its own label to "Copied" for ~1.2s, then reverts to "Copy path" — same component/behavior reused for both the web-zip line and each desktop platform's success row.
- "Reveal in folder" button correctly absent in this browser-only dev environment (`hearthNative()` returns `null` when `window.hearthNative` is undefined) — expected, Electron-only.
- Primary button label swaps "Export" → "Exporting…" while busy, and the secondary button swaps "Cancel" → "Close" once a result exists; both correctly `disabled` while busy.
- **Web pane fully resets on each dialog (re)open**: since `Modal` unmounts its children while closed (`{open && (...)}`), `WebPane` remounts fresh — format reverts to "Web build folder", zip unchecks, and any previous result/error clears. Confirmed this is a full remount, not a stale-state bug: re-opening after a web export leaves the Desktop-job-driven `reopenMode()` routing untouched (still lands on Web, since web exports don't touch the desktop job store).
- **Error path** (web): output dir under a `chmod 000` folder → clear, readable error in `.export-errors[role=alert]` (`exportWeb failed: EACCES: permission denied, mkdir '.../readonly-out/nested'`), and the same message is mirrored into the Console panel as an error-level log entry (badge count on the Console tab incremented) — consistent with the rest of the editor's command-logging convention.
- Long output-directory path (~200 chars): result panel wraps the path across multiple lines (`word-break: break-all`) with no horizontal overflow of the dialog — Wave J's "long-path overflow" checklist item confirmed clean.

**Desktop pane**
- Signing status line renders `<SigningIcon/>` + text; in this ad-hoc-signing dev environment confirmed "Ad-hoc signing" (from `GET /api/export/capability` → `{mode: "adhoc"}`). `identity` / `identity+notarize` label variants are code-correct (`signingStatusLabel`) but not visually exercised — no signing identity available in this environment (see Not covered).
- Platform grid: all 4 offered platforms render (`macOS (Apple Silicon)`, `macOS (Intel)`, `Windows`, `Linux`), all checked by default (`defaultPlatformSelection`), each independently toggleable, `.check-row-compact:has(input:checked)` accent styling confirmed.
- Zero-platforms-selected validation: unchecking all four shows "Pick at least one platform." (`.field-note`, warn color) and disables the primary Export button; re-checking exactly one re-enables it.
- **Real desktop export, single platform (darwin-arm64, using the locally-cached Electron zip)**: full lifecycle observed live over the ws — row starts, badge "QUEUED"→"BUILDING", stage label "Packaging" with message "packaging darwin-arm64", then "DONE · ad-hoc" badge with the zip path (`export/desktop/mini_platformer-darwin-arm64.zip`) and app bundle on disk (`Mini Platformer-darwin-arm64/Mini Platformer.app`). Primary button label progression confirmed: "Export" → "Building…" → "Export again"; secondary button "Cancel" (disabled while running) → "Close".
- **Real desktop export, two platforms (darwin-arm64 + darwin-x64)**: confirmed simultaneous-looking-but-sequential build order via badges — one row "BUILDING" while the other shows "QUEUED" (grey `badge-pending`), both eventually reach "DONE · ad-hoc". Ember-accent active row (`export-row.active`: full orange border + soft tint) confirmed to track exactly the currently-building platform, not the queued one — Wave J checklist item confirmed.
- **Per-platform error path**: pointed the output dir at a `chmod 000` folder, ran a darwin-arm64-only export → row correctly lands on `status-error` (red border/background), badge "FAILED", and the row body shows the real underlying error text (`exportDesktop failed: packaging darwin-arm64 failed: EACCES: permission denied, mkdir '.../darwin-arm64'`). Primary button correctly offers "Export again" (not stuck on "Building…").
- **Close-during-export lock**: verified with a mouse — the Cancel button and the segmented control (`disabled={jobRunning}`) are both correctly disabled and inert while a desktop job runs (see EXPORTDIALOG-1 for the Escape-key gap in this same guard).
- **Reopen-after-completion persistence**: closed the dialog (Escape) after a finished single-platform job, reopened — `reopenMode()` correctly routes back to the Desktop pane and the finished row (badge, zip path, Copy-path button) is still there, matching Wave J's updated checklist expectation ("reopen-after-completion to SHOW results," not wipe them).
- **Project-switch reset**: used "Close project" → reopened the same project path (a fresh `connectWs()` per project open) → Export dialog defaults to the Web pane and the Desktop pane's progress list is empty (0 rows), confirming `resetExportJob()` at `connectWs`/`disconnectWs` (`store.ts:615,680`) correctly wipes a prior project's job state rather than leaking it into the newly-opened project.
- Output-directory field for Desktop behaves the same as Web's (free text, Enter-to-submit, disabled while running).

## Not covered
- **Windows / Linux desktop builds**: not run for real (per dispatch: real desktop exports were opt-in only where a local Electron cache exists; only `darwin-arm64`/`darwin-x64` zips were present under `~/Library/Caches/electron`, so only those two platforms were built). `win32-x64`/`linux-x64` rows are exercised only up through "QUEUED" state via the two-platform run, never through an actual packaging step.
- **Signing capability states other than "adhoc"**: `identity` and `identity+notarize` signing-status copy (and the corresponding "signed"/"notarized" row-badge sub-labels) are verified by reading `signingStatusLabel`/`RowBadge` but not observed live — this dev machine/environment has no Developer ID signing identity configured for `@hearth/shipping` to select.
- **Platform grid in a light theme**: N/A — per Wave J notes in `progress.md` ("Editor is single-theme dark — light-theme checklist items N/A"), the editor ships dark-only; no light-theme variant exists to check.
- **"Reveal in folder" button**: code-verified to correctly no-op/hide in this browser-dev environment (`window.hearthNative` undefined); its actual OS-level reveal behavior can only be exercised inside a packaged Electron build, out of scope here.
- **Full keyboard Tab-order audit inside the dialog**: relied on the browser's native `<dialog>`/`showModal()` focus trap (standard, not custom code) rather than manually walking every Tab stop; spot-checked only the segmented control's roving tabindex and Escape-to-close.
