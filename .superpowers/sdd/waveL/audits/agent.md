# Audit: agent (example: ember-horde, port 5221)

## Findings

### AGENT-1 · defect · high
- Element: Launcher primary button (`Start agent` / `Open Terminal`) in `.agent-toolbar`, `AgentPanel.tsx`
- Observed: The button's only `disabled` condition is `!projectPath` (`disabled={!projectPath}` in `AgentPanel.tsx`) — there is no `disabled={running}` guard. Reproduced live: start a plain shell (`Terminal / other CLI` → `Open Terminal`), confirm status reads "Running shell". While it's running, switch the launcher `<select>` to `Claude Code` (or `Codex`) without stopping anything first. The button relabels to `Start agent`, stays bright orange, and `isDisabled()` returns `false` — fully clickable — while `Stop` is *also* still enabled and the status still says "Running shell". Per `ws.ts`'s `startPty`, a new `pty-start` unconditionally kills whatever pty the project root already has and re-homes ownership before spawning the new command. So clicking `Start agent` in this state would silently kill the running shell session and spawn the real `claude` CLI (or `codex`) with zero confirmation. Both `claude` and `codex` are installed on PATH in this environment, so this is not a hypothetical — it's the exact "hits Start agent by muscle memory, kills what I had running, launches a real paid session" scenario the audit brief calls out as a cost risk. (Verified the disabled-state gap only; per this audit's constraint I did not actually click through to confirm the spawn.)
- Expected: The primary action button (and the launcher/mode selects) should be disabled — or at least gated behind a confirmation — whenever `agent.session.status === 'running'`, so switching launchers mid-session can't silently supersede a live process.

### AGENT-2 · defect · med
- Element: `Restore checkpoint` button (`Timeline.tsx`) vs. the Timeline's own live activity feed
- Observed: `Restore checkpoint`'s disabled state is bound to `diff?.hasChanges`, and `diff` is only recomputed by an explicit `refreshDiff()` call — never automatically when a new journal entry lands. Confirmed via `store.ts`: the WS `journal` frame handler calls `pushJournalFeed()` (which the Timeline renders immediately) and, for external entries, `refresh()` — but `refresh()` does not touch `diff`; only `Review changes` (`requestDiffFocus` → `Workspace.tsx`'s effect) or the Changes panel itself call `refreshDiff()`. Live repro: took a Checkpoint, then ran `hearth create entity Arena X` from a separate shell. The new `createEntity` row appeared in the Timeline within ~2s (live, correct), but `Restore checkpoint` stayed disabled until `Review changes` was clicked — at which point it correctly enabled.
- Expected: A user watching the Timeline fill up with an agent's work and wanting to immediately revert souldn't find `Restore checkpoint` inexplicably greyed out. Either refresh `diff` whenever an external journal entry lands (same trigger that already updates the Timeline), or don't visually imply the two are in sync.

### AGENT-3 · friction · med
- Element: Permission-mode `<select>` (Read-only / Safe edit / Full / All) in `.agent-toolbar`
- Observed: The mode picker is fully enabled and shows mode-specific tooltip/hint text regardless of which launcher is selected, but it is only ever consumed when launcher === `claude` (`startAgent()` only calls `apiPrepareAgent(...)` in that branch). For `codex` and `shell`, the selected mode is passed along on the `pty-start` WS frame (`useAgentSocket.ts`'s `start(command, mode)`) purely "so the server sees what was requested" per its own doc comment — but `ws.ts`'s `startPty(root, ws, command)` never accepts or reads a `mode` argument at all. So for 2 of the 3 launcher options, changing the mode dropdown has literally zero effect, with no visual indication that it's inert.
- Expected: Either disable/hide the mode picker for non-Claude launchers, or make it visibly informational only when it doesn't do anything, so it doesn't imply a permission scope that isn't actually applied.

### AGENT-4 · friction · med
- Element: "Manual setup" → `MCP: Claude Code` and `MCP: generic clients` `CodeBlock`s
- Observed: Both example blocks are built from static templates (`mcpClaudeBlock`, `mcpJsonBlock` in `AgentPanel.tsx`) that only ever include `--project <path>` — neither references `agentMode` / includes a `--mode` flag, even though the panel's own Mode picker is sitting directly above and the "automatic" Prepare flow (verified directly against `/api/agent/prepare`) does write `--mode <tier>` into `.mcp.json` for whichever tier is selected. A user who picks "Full (no build)" in the dropdown, then copies the Manual Setup command instead of clicking Start agent, gets a config that's implicitly read-only (the server's own default when `--mode` is omitted) with no on-screen indication the selected tier didn't carry over.
- Expected: The Manual Setup code blocks should reflect the currently-selected permission mode (append `--mode <tier>`), the same way `cliBlock`/`manualProjectPath` already reflect live panel state.

### AGENT-5 · friction · low
- Element: Timeline empty state ("No activity yet")
- Observed: `journalWatcher.ts` deliberately baselines `lastDelivered` to the on-disk journal's *current* `lastSeq()` the moment a project's watcher starts — by design, it never backfills history, only delivers entries appended after that point (confirmed in code and by observation: `journalFeed` was `[]` on every fresh project open in this audit, despite the project's on-disk journal having real history from earlier CLI runs in the same session). This is a reasonable design (matches the terminal's own per-epoch scrollback semantics) but the copy "No activity yet" doesn't communicate the "this session only" scope — it reads as "nothing has happened in this project," which isn't true.
- Expected: Copy like "No activity yet this session" (or similar), so the empty state doesn't imply the project itself is untouched.

### AGENT-6 · friction · low
- Element: Terminal scrollback cap (`SCROLLBACK_CAP_BYTES` = 200KB, `useAgentSocket.ts`)
- Observed: `droppedBytes` is tracked precisely in `AgentSocketState` for exactly this purpose, but it is never rendered anywhere in the UI (`Terminal.tsx`/`AgentPanel.tsx` never reference it). A long-running session that exceeds 200KB of output silently loses its earliest lines with no on-screen cue that truncation happened (not exercised live to the full cap in this audit — confirmed by code path only, since the eviction logic is in `appendCapped`).
- Expected: Surface `droppedBytes` somewhere near the terminal (e.g. a small "n KB of earlier output trimmed" note) when it's non-zero, so a user scrolling up for something isn't left wondering whether they missed it or it's just gone.

### AGENT-7 · friction · low
- Element: `Checkpoint` button — Agent Timeline (`Timeline.tsx`) vs. Toolbar (`Toolbar.tsx`) / Diff panel (`DiffPanel.tsx`)
- Observed: All three call the identical `exec('snapshotProject', {}, { quiet: true })`, but only `Toolbar.tsx` and `DiffPanel.tsx` follow it with `log('info', 'command', 'Checkpoint saved. The Changes panel now compares against this checkpoint.')`. The Agent Timeline's `snapshot()` calls `exec(...)` and nothing else — confirmed live: clicking Checkpoint from the Agent panel shows the ✓ badge next to the button but produces no Console feedback, unlike the same action from the main toolbar.
- Expected: Same action, same feedback, regardless of entry point — add the matching `log()` call to the Timeline's `snapshot()`.

### AGENT-8 · friction · low
- Element: Timeline row labels for property-mutation commands
- Observed: `entryToRow`'s label is `entry.summary || entry.command`; for `setComponentProperty` the row rendered as `setComponentProperty` with meta text `Arena` (the scene name) — not which entity or property changed. Live repro: ran `hearth set Arena AuditProbeEntity Transform.position.x 250`; the resulting Timeline row gave no visual indication it was `AuditProbeEntity`'s `Transform.position.x` that changed — a user would need to open the Diff panel to find out. This is exactly the surface meant to build trust in what an unattended agent is doing, and its most common mutation type is also its least legible one.
- Expected: Include the entity name (and ideally the property path) in the row label/meta for `setComponentProperty` and similar per-entity mutations, not just the scene.

### AGENT-9 · polish · low
- Element: Agent panel default docked height (`.agent-body`, bottom dockview group)
- Observed: On first opening a project, the Agent panel's bottom-group docked height is quite short — measured terminal pane at 143px tall (≈6 visible rows) and the Timeline side-rail at the same height, both requiring the user to manually drag the horizontal sash upward (confirmed this works — dragging enlarged the terminal pane to 445px) to get a usable amount of vertical space. Given the terminal is one of this surface's two primary, high-attention elements, the default is cramped enough to feel like a bug on first look.
- Expected: A taller default initial height for the bottom panel group (or specifically the Agent panel) so the terminal is usably sized without requiring the user to discover the resize handle first.

### AGENT-10 · polish · low
- Element: `Checkpoint` button tooltip — Agent Timeline vs. Toolbar
- Observed: The main toolbar's `Checkpoint` button title reads "Save a checkpoint you can review and restore (⇧⌘S)"; the Agent Timeline's own `Checkpoint` button (identical action, per AGENT-7) has title "Save a checkpoint you can review and restore" — same copy, minus the shortcut hint.
- Expected: Either both mention the shortcut, or neither — as-is it reads as an oversight rather than a deliberate difference.

## Verified working
- Launcher `<select>` (Claude Code / Codex / Terminal-other-CLI): switching options correctly updates the hint text, the primary button label (`Start agent` vs `Open Terminal`), and (per AGENT-3) has no effect on the mode picker's own display.
- Permission-mode `<select>` (Read-only / Safe edit / Full / All): all 4 tiers show correct, distinct `title` tooltips and hint text matching `PERMISSION_DOCS` composition (`full` = safe-edit+code-edit+asset-edit; `all` = every tier including build).
- `/api/agent/detect`: correctly reports `found`/`version` for both `claude` and `codex` when present on PATH.
- Detect states via mocked responses: both-not-found → "Install Claude Code" (Claude) / "Codex not found" (Codex), both correctly disabled-looking and non-crashing; network failure on detect → "Couldn't check for Claude Code" + working `Re-detect`; recovering the mock and re-clicking `Re-detect` correctly restores "Start agent".
- Shell launcher (`Terminal / other CLI`) stays enabled and reachable (`Open Terminal`) regardless of Claude/Codex detection results — never blocked by CLI-detection failures.
- `POST /api/agent/prepare` (the actual `.mcp.json` write, exercised directly to avoid spawning the CLI): correctly merge-writes `mcpServers.hearth` with `--project`/`--mode` for all 4 modes (`read-only`, `safe-edit` → `safe-edit`, `full` → `safe-edit,code-edit,asset-edit`, `all` → `all`); correctly errors on an unknown mode, a missing `project` field, a path with no `hearth.json`, and an existing-but-corrupt `.mcp.json` (refuses to overwrite, clear message); re-running with the same args is idempotent.
- Opening a plain shell terminal (`Terminal / other CLI` → `Open Terminal`) spawns a real PTY (confirmed real zsh prompt, correct `cwd` = project root).
- Typing into the terminal (`echo`, multi-line `seq 1 200`) round-trips correctly; mouse-wheel scroll up/down through scrollback works.
- Terminal reattach: starting a shell, typing a unique marker, switching to a different dockview tab (Console) and back to Agent reattaches with the **full scrollback intact** (verified via DOM text diff, marker preserved) and the session's `running` status preserved — the documented "hiding the panel must not kill the pty, and reopening replays scrollback" contract holds.
- Session status label lifecycle: `Idle` → `Running shell` (green) → `Exited` (grey, after Stop) → `Idle` again after the WS connection drops (`resetAgentSocket`); `Stop` correctly disabled while idle, enabled while running.
- Panel resize: dragging the horizontal sash above the bottom panel group correctly enlarges the terminal pane (143px → 445px measured) with no layout breakage.
- Timeline live updates from **external** CLI activity (run from a separate shell, not the embedded terminal): `createEntity`, `setComponentProperty`, `validateProject` (with correct "0 errors, N warnings" meta), and a deliberately-failing `setComponentProperty` (entity not found) all appeared within ~2s with correct icon, label, ok/error styling, and relative timestamp — confirming the on-disk-journal-watcher path (CLI or MCP agent → journal file → `journalWatcher.ts` → WS `journal` frame → Timeline) works end-to-end without the embedded terminal needing to be involved at all.
- Checkpoint → further changes → Review changes → Restore checkpoint full round-trip: verified via CLI inspection afterward that the reverted project state genuinely matched the checkpoint baseline (extra entity created after the checkpoint was gone).
- Restore-checkpoint confirm dialog: correct title/body copy ("Restore checkpoint?" / full discard-warning body), `Cancel` genuinely cancels with zero project mutation (re-verified via CLI), and reopening + confirming after a prior Cancel correctly reverts (isolated, clean repro — no residual effect from the Cancel).
- Manual Setup accordion: expand/collapse toggling works (chevron rotates via `aria-expanded`), and all 3 `CodeBlock`s render the expected commands with the project's real resolved paths substituted in.
- Copy buttons (all 3 in Manual Setup): each correctly writes its block's exact text to the OS clipboard (verified via `navigator.clipboard.readText()`) and shows a "Copied" label that reverts to "Copy" after ~1.2s.
- Permission table (5 rows) and golden-rules list (5 items) render under Manual Setup.
- WS reconnect: killing the dev server mid-session (own PID only) immediately flips session status to `Idle` and the connection indicator to `ws-dot-disconnected`; restarting the server on the same port, the client's exponential-backoff reconnect logic reconnects on its own (no page reload needed) and a subsequent `refresh()` repopulates the Hierarchy/project state correctly.

## Not covered
- Never actually started the real `claude` or `codex` CLI, per this audit's explicit cost/session-risk constraint — the launch button's happy path (successful spawn, live output from a real agent) was verified only up to the `.mcp.json`-prepare step (tested directly against `/api/agent/prepare`) and via source review of `startAgent()`/`ws.ts`'s `startPty`, not by clicking through.
- "Install Claude Code" button: not clicked, since (per source) it starts a real shell and immediately types+submits `npm install -g @anthropic-ai/claude-code`, which would perform a real, time-consuming global npm install as a side effect. Verified only via code reading (`installClaude()` in `AgentPanel.tsx`).
- Scrollback eviction at the full 200KB cap was not exercised live (would require generating hundreds of KB of terminal output); the cap/eviction/`droppedBytes` logic (AGENT-6) was confirmed by code reading only.
- OS-level paste-into-terminal (e.g. system clipboard → xterm via Cmd+V) was not exercised; only typed input and the Manual Setup's copy-out buttons were tested.
- Multi-socket / two-tabs-same-project pty ownership handoff (`ws.ts`'s "a new pty-start re-homes ownership, the stale socket stops receiving output") was reviewed in source but not exercised with two concurrent browser contexts.
- Global keyboard shortcut for Checkpoint (⇧⌘S, see AGENT-10) was tested with the embedded terminal both focused and unfocused and did not appear to fire in either case in headless Chrome — but this is inconclusive and not filed as a finding: real macOS Chrome may intercept Cmd+Shift+S natively (e.g. as "Save Page As") before it reaches page JS, which would produce the same symptom without an app bug. Not reproducible in a way I could confidently attribute to Agent-surface code.
- An intermittent, non-Agent-surface issue surfaced repeatedly during setup: opening a project occasionally throws an uncaught `Cannot read properties of null (reading 'clear')` (consistent with `Workspace.tsx`'s `buildDefaultLayout`'s `api.clear()` running against a not-yet-ready dockview api) which leaves the toolbar's `View` menu button disabled/unresponsive until the page is reloaded. This is a Workspace/dockview-level race, not part of the Agent panel itself, so it isn't filed as an AGENT finding — noting it here since it repeatedly interfered with this audit's "toggle the panel off/on via View menu" workaround and is worth another surface's auditor being aware of.
