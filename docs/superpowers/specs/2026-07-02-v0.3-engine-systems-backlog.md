# Hearth v0.3+ — Engine Systems Backlog

Captured 2026-07-02 from Jake's direction while v0.2 was in flight. These are
the systems that take Hearth from "capable" to "full engine". They were
deliberately deferred so v0.2 (dockable workspace, game UI, polygon
colliders, audio, web export) lands as a solid, tested milestone.

**The standing rule for every item here: agent-native first.** Each system
ships as schemas + commands before editor UI — inspectable via
`hearth ... --json`, exposed as MCP tools, and testable in headless
playtests — so a coding agent can understand and build out games in full,
not just humans clicking the editor. Editor UI comes second and calls the
same commands.

**The second standing rule (Jake, 2026-07-02): no engine chrome in shipped
games.** Like Godot, every visible part of a game is the user's work. The
v0.2 web export's Hearth-themed click-to-start screen violates this — it
was a stopgap for the browser audio-gesture policy. The end state: an
exported game boots straight into the user's initial scene; a start screen
or menu, if the game has one, is just a scene the user built (UIElement
buttons + a scene-switching API); the audio context resumes silently on the
first natural input instead of demanding a branded gate; loading visuals
(background color, optional image, spinner on/off) come from buildSettings
with neutral defaults and zero Hearth branding. Nothing the player sees
should be authored by the engine.

## Already covered (for orientation)

Common elements (Vec2/color in schemas), file system (FsLike), collision
(box/circle, polygon SAT in v0.2), window management (Electron shell +
launcher), input handling, text rendering, textured/untextured quads
(SpriteRenderer), world/scene management, entities, script lifecycle hooks,
UI widgets v1 (UIElement, v0.2), audio playback (v0.2), asset import
(`importAsset` command exists; editor drag-drop UX lands in v0.2 wave 2).

## 1. Rendering pipeline v2

- **2D lighting**: `Light2D` component (radius, color, intensity, enabled) +
  `Camera.ambientLight` (0–1); Pixi darkness overlay with additive lights.
  Quads then render "with lighting" for free.
- **Polylines / beams**: `LineRenderer` component (points, width, color,
  closed, layer) — doubles as debug beams and simple vector art.
- **Debug draw layer**: toggleable overlay rendering colliders (boxes /
  circles / polygons), grids, velocity vectors; `hearth screenshot --debug`
  once screenshots exist so agents can see hitboxes too.
- **Particle systems + streaks/trails**: `ParticleEmitter` component
  (rate, burst, lifetime, speed/spread, gravity, start/end color+size,
  seed — deterministic so headless runs can assert particle counts); trail
  preset for streaks.
- Command surface: all are components, so existing create/set/inspect
  commands cover them; add `COMPONENT_DOCS` entries + validation.

## 2. Sprite animation playback

`SpriteAnimator` component (animation asset id, fps, playing, loop) cycling
`SpriteRenderer.assetId` through the animation asset's frames. Animation
assets already exist (`createAnimationAsset`). Script API:
`ctx.getComponent('SpriteAnimator')` mutation + a `play(name)` convenience.

## 3. Physics engine v2

- `PhysicsBody` gains mass, restitution (bounciness), friction.
- Collision layers/masks; one-way platforms; circle-accurate resolution.
- Longer term: impulse solver, joints. Runtime stays deterministic
  (fixed timestep) — that's Hearth's playtest + future-multiplayer story.

## 4. Script standard library ("common elements")

- `ctx.math`: vec2 ops (add/sub/scale/dot/length/normalize/distance/lerp/
  angle), clamp, color helpers (hex↔rgb, lerp). Matrix (2×3) if/when
  parenting transforms need it.
- `ctx.events`: global pub/sub — `emit(name, data)`, `on(name, fn)` with
  per-entity auto-cleanup, plus an `onEvent(ctx, name, data)` hook.
  Headless run report records emitted events so playtests can assert them.

## 5. Pathfinding

Grid A* over Tilemap solids (+ static colliders rasterized to the grid):
`ctx.scene.findPath(from, to) -> Vec2[] | null`. CLI/MCP:
`hearth inspect path <scene> --from x,y --to x,y --json` so agents can
reason about reachability without running a scene.

## 6. Asset pipeline v2

- Spritesheet import + slicing; imported-image color/size metadata.
- Music (long-form audio) streaming — current Web Audio path decodes whole
  buffers, fine for SFX, wasteful for music.
- Font assets for Text.
- Editor: thumbnails, previews, bulk import (drag-drop UX itself ships in
  v0.2 wave 2 via `importAsset` + an upload route).

## 7. UI widgets v2

Layout containers (anchor groups / vertical-horizontal stacks), sliders,
toggles, focus + keyboard navigation. Keep composing with Text /
SpriteRenderer rather than a parallel widget tree.

## 8. Scripting v2 — TypeScript, complete at boot

**Language decision: TypeScript.** Plain JavaScript stays valid forever (TS
is a superset), but TS is the first-class language. Why:

- The engine, editor, and web player are already TypeScript/JS — shipped
  games need no extra VM (Lua would mean a WASM interpreter in every
  export; a custom GDScript-like language is the worst option: agents can't
  write it reliably, and Hearth's thesis is agent-written game logic).
- Coding agents (and their training data) are strongest in TS/JS — typed
  APIs turn agent mistakes into compile errors with messages agents can
  act on, instead of silent runtime bugs.
- A generated `hearth.d.ts` (typed `ctx`, components, events) gives humans
  editor autocomplete and gives agents an inspectable API contract
  (`hearth inspect api --json`).

What "fully complete and ready to go when the engine boots" means:

- **Compile step**: esbuild/sucrase transform on load (editor, CLI, player
  build) — TS + ES module syntax, imports between scripts (shared utility
  modules in `scripts/lib/`), no more Function-eval restrictions. Structured
  diagnostics from `hearth validate` (file, line, message) so agents can
  fix their own type errors.
- **Complete stdlib at boot** — no half-wired APIs: scene management
  (`ctx.scenes.load(id)`, transitions — this is also what makes user-built
  menus/start screens possible), timers + tweens, seeded RNG, persistence
  (`ctx.save`/`ctx.load` for save games), camera control, ctx.math,
  ctx.events, ctx.audio, pathfinding — everything documented in the
  generated AGENTS.md and typed in hearth.d.ts.
- Script templates, docs, and examples move to TS; `hearth create script`
  emits `.ts` by default.

## 9. Visual logic editor (research)

Node-based logic that round-trips to the same command system + generated
scripts, so agents and humans edit the same artifact. Big; needs its own
design pass. Scripting (JS) remains the source of truth until then.

## Suggested wave grouping for the next run

- **Wave 0 (foundation, sequence first)**: scripting v2 — the TS compile
  step, module loader, and scene-management API touch the runtime, player,
  and core at once, and most other waves build on the new ctx surface.
  Include the no-chrome export rework here (kill the branded start screen,
  buildSettings-driven loading, silent audio unlock).
- **Wave A (runtime-heavy)**: rendering v2 (lighting, lines, debug,
  particles) + sprite animation — one agent owning packages/runtime.
- **Wave B (core+script API)**: math/events stdlib, pathfinding, physics v2
  properties + validation — parallel with A only if file ownership is split
  (stdlib/pathfinding are new modules; physics touches the same files as A's
  debug draw, so sequence those).
- **Wave C**: asset pipeline v2 + editor UX + docs/examples.
