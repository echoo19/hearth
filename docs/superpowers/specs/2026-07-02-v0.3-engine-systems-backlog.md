# Hearth v0.3+ — Engine Systems Backlog

Captured 2026-07-02 from Jake's direction while v0.2 was in flight. These are
the systems that take Hearth from "capable" to "full engine". They were
deliberately deferred so v0.2 (dockable workspace, game UI, polygon
colliders, audio, web export) lands as a solid, tested milestone.

**The standing rule for every item here: agent-native first.** Each system
ships as schemas + commands before editor UI — inspectable via
`hearth ... --json`, exposed as MCP tools, and testable in headless
playtests — so a coding agent can understand and build out games in full,
not just humans clicking the editor. Editor UI comes second and calls the
same commands.

**The second standing rule (Jake, 2026-07-02): no engine chrome in shipped
games.** Like Godot, every visible part of a game is the user's work. The
v0.2 web export's Hearth-themed click-to-start screen violates this — it
was a stopgap for the browser audio-gesture policy. The end state: an
exported game boots straight into the user's initial scene; a start screen
or menu, if the game has one, is just a scene the user built (UIElement
buttons + a scene-switching API); the audio context resumes silently on the
first natural input instead of demanding a branded gate; loading visuals
(background color, optional image, spinner on/off) come from buildSettings
with neutral defaults and zero Hearth branding. Nothing the player sees
should be authored by the engine.

## Already covered (for orientation)

Common elements (Vec2/color in schemas), file system (FsLike), collision
(box/circle, polygon SAT in v0.2), window management (Electron shell +
launcher), input handling, text rendering, textured/untextured quads
(SpriteRenderer), world/scene management, entities, script lifecycle hooks,
UI widgets v1 (UIElement, v0.2), audio playback (v0.2), asset import
(`importAsset` command exists; editor drag-drop UX lands in v0.2 wave 2).

## 1. Rendering pipeline v2

- **2D lighting**: `Light2D` component (radius, color, intensity, enabled) +
  `Camera.ambientLight` (0–1); Pixi darkness overlay with additive lights.
  Quads then render "with lighting" for free.
- **Polylines / beams**: `LineRenderer` component (points, width, color,
  closed, layer) — doubles as debug beams and simple vector art.
- **Debug draw layer**: toggleable overlay rendering colliders (boxes /
  circles / polygons), grids, velocity vectors; `hearth screenshot --debug`
  once screenshots exist so agents can see hitboxes too.
- **Particle systems + streaks/trails**: `ParticleEmitter` component
  (rate, burst, lifetime, speed/spread, gravity, start/end color+size,
  seed — deterministic so headless runs can assert particle counts); trail
  preset for streaks.
- Command surface: all are components, so existing create/set/inspect
  commands cover them; add `COMPONENT_DOCS` entries + validation.

## 2. Sprite animation playback

`SpriteAnimator` component (animation asset id, fps, playing, loop) cycling
`SpriteRenderer.assetId` through the animation asset's frames. Animation
assets already exist (`createAnimationAsset`). Script API:
`ctx.getComponent('SpriteAnimator')` mutation + a `play(name)` convenience.

## 3. Physics engine v2

- `PhysicsBody` gains mass, restitution (bounciness), friction.
- Collision layers/masks; one-way platforms; circle-accurate resolution.
- Longer term: impulse solver, joints. Runtime stays deterministic
  (fixed timestep) — that's Hearth's playtest + future-multiplayer story.

## 4. Script standard library ("common elements")

- `ctx.math`: vec2 ops (add/sub/scale/dot/length/normalize/distance/lerp/
  angle), clamp, color helpers (hex↔rgb, lerp). Matrix (2×3) if/when
  parenting transforms need it.
- `ctx.events`: global pub/sub — `emit(name, data)`, `on(name, fn)` with
  per-entity auto-cleanup, plus an `onEvent(ctx, name, data)` hook.
  Headless run report records emitted events so playtests can assert them.

## 5. Pathfinding

Grid A* over Tilemap solids (+ static colliders rasterized to the grid):
`ctx.scene.findPath(from, to) -> Vec2[] | null`. CLI/MCP:
`hearth inspect path <scene> --from x,y --to x,y --json` so agents can
reason about reachability without running a scene.

## 6. Asset pipeline v2

- Spritesheet import + slicing; imported-image color/size metadata.
- Music (long-form audio) streaming — current Web Audio path decodes whole
  buffers, fine for SFX, wasteful for music.
- Font assets for Text.
- Editor: thumbnails, previews, bulk import (drag-drop UX itself ships in
  v0.2 wave 2 via `importAsset` + an upload route).

## 7. UI widgets v2

Layout containers (anchor groups / vertical-horizontal stacks), sliders,
toggles, focus + keyboard navigation. Keep composing with Text /
SpriteRenderer rather than a parallel widget tree.

## 8. Scripting v2 — Lua first-class, complete at boot

**Language decision (revised by Jake, 2026-07-02): Lua is the first-class
scripting language; JavaScript stays supported; Python is ruled out.**

Why Lua:

- It is the embedded game-scripting standard (Roblox/Luau, LÖVE, Defold,
  Pico-8, WoW mods) — designed to be dropped into engines, and it gives
  Hearth a real on-ramp for the Roblox generation.
- Beginner-intuitive for humans (`if score > 10 then ... end`; no braces,
  no `this`, no async footguns) while agents write it well — the
  Roblox/modding corpus makes LLMs solid at Lua.
- Tiny in shipped games: a browser Lua VM (wasmoon, Lua 5.4 over WASM) is
  ~250KB. Python was rejected for exactly this: browser Python means
  bundling Pyodide (~15MB WASM) into every exported game with poor
  per-frame performance — Python is not an embeddable game language.
- Sandboxed by design (good story for running agent-written code) and
  deterministic once `os.time`/`math.random` are replaced with the
  engine's seeded equivalents — same VM in the editor, headless playtests
  (Node), and the web player.
- A custom GDScript-like language stays ruled out: agents can't write what
  they weren't trained on.

Shape of the work:

- One `ctx` API, two bindings: the existing JS path stays (it's what the
  runtime natively speaks — zero cost to keep, right for web devs), Lua
  becomes the default: `hearth create script` emits `.lua`; templates,
  docs, examples, and AGENTS.md teach Lua first.
- Mitigate the lost compile-time types: Luau-style optional type
  annotations if cheap, and structured diagnostics from `hearth validate`
  (file, line, message — including a validate-time dry execution of
  lifecycle hooks) so agents can self-fix errors without booting the game.
- Editor support built out over time: Lua syntax highlighting,
  completion for the ctx API, inline diagnostics.
- **Complete stdlib at boot** — no half-wired APIs, identical across both
  languages: scene management (`ctx.scenes.load(id)`, transitions — also
  what makes user-built menus/start screens possible), timers + tweens,
  seeded RNG, persistence (`ctx.save`/`ctx.load` for save games), camera
  control, ctx.math, ctx.events, ctx.audio, pathfinding — everything
  documented in the generated AGENTS.md and inspectable via
  `hearth inspect api --json`.

## 9. Embedded agent panel ("make me this game: [spec]")

Jake, 2026-07-02: the editor should host an agent directly — a chat panel
where a user pastes a game spec and an agent plans and builds it, fully and
accurately. This evolves the old "no built-in AI calls" non-goal rather
than contradicting it: the engine core stays offline and model-free; the
panel is an **optional, bring-your-own-agent adapter**. Nothing local is
gated (business model rule); a managed zero-setup agent is the natural
Hearth Cloud feature later.

**Auth rule (Jake, 2026-07-02): never force API credits on people who
already pay for a subscription.** The default adapter is the user's own
installed Claude Code — Hearth spawns it against the project, so it runs
under their existing claude.ai login and Pro/Max plan, no API key ever
entered. API-key adapters (Agent SDK in-process, OpenAI-compatible
endpoints) are the fallback for users without Claude Code or who want
other models.

Why this is cheap for Hearth and hard for everyone else: the agent needs no
special powers. Editor, CLI, and MCP already drive the same command
registry; every mutation is schema-validated; snapshot/diff/revert is the
approval mechanism; playtests are the agent's self-verification loop.

Architecture:

- **Login lives in the panel (Jake, 2026-07-02 — low friction is the
  requirement).** First run: detect installed agent CLIs (claude, codex) →
  one-click install if missing → run the CLI's own OAuth login inside the
  panel (browser pops, user signs into their claude.ai / ChatGPT account,
  credentials persist on the machine) → never asked again. No API keys, no
  config files, no copy-paste anywhere in the happy path.
- **V1 is an integrated terminal, not a chat UI**: embed a real PTY
  (node-pty + xterm.js in Electron) running the agent CLI's own TUI,
  already cd'd into the project with hearth MCP pre-wired. Full agent
  capability on day one, the exact tool users already know, tiny build
  cost — and login flows work in it natively. The structured chat UI
  layers on top later.
- **Adapter interface in the project server** (Electron main / vite dev
  server both get it): `AgentBackend { send(messages, tools) -> stream }`.
  - **Primary adapter: spawn the user's installed Claude Code CLI** against
    the project (interactive PTY for v1; stream-json headless for the v2
    chat UI) — subscription auth preserved, zero key management.
  - Codex CLI supported the same way (ChatGPT-account login).
  - Fallback adapter: Claude Agent SDK in-process with an API key, tools
    bound straight to the command registry (no MCP hop).
  - Keep the interface model-agnostic (OpenAI-compatible endpoints too).
- **Onboarding collapse**: once the panel ships, the website and first-run
  story become "download Hearth → open it → click Connect → describe your
  game." The MCP/CLI wiring instructions move off the critical path to an
  "use Hearth from your own terminal/agent" page for external workflows.
- **Legal line (verify against current Anthropic terms before building)**:
  hosting the user's genuine, unmodified CLI in an embedded terminal is
  fine — same as VS Code/tmux hosting Claude Code; we never touch
  credentials or proxy traffic. What is NOT fine on subscription auth:
  wrapping headless mode in our own chat UI (third-party products are
  expected to bill via API), reusing/extracting OAuth tokens, or spoofing
  client IDs — tools that did this got blocked. Therefore: v1 terminal =
  subscription-friendly; v2 custom chat UI = API-key (Agent SDK) only,
  unless Anthropic explicitly OKs subscription use (worth asking —
  partnership is plausible for a tool launching real Claude Code).
  Trademark: descriptive "works with Claude Code" is fine; no logos or
  implied endorsement.
- **System prompt = what we already generate**: AGENTS.md + component docs
  + command docs + the golden rules. The panel enforces the same loop:
  snapshot → inspect → build → validate → playtest → diff.
- **Trust UX (the differentiator)**: a live timeline of every structured
  command the agent runs (createEntity Coin → attachScript → runPlaytest ✓)
  instead of opaque file edits; permission-mode selector up front
  (read-only through build); Approve / Revert buttons wired to the existing
  snapshot + diff; playtest results inline; cost/token readout.
- **"Make me this game" flow**: spec → agent drafts a build plan (scenes,
  entities, scripts, assets, playtests) shown for one approval → executes
  with progress → runs validate + playtests until green → presents the
  diff and a playable Game panel. The user's first experience of Hearth
  can be typing a paragraph and playing the result.

## 10. Visual logic editor (research)

Node-based logic that round-trips to the same command system + generated
scripts, so agents and humans edit the same artifact. Big; needs its own
design pass. Scripting (JS) remains the source of truth until then.

## 11. Gamepad input

Map the browser Gamepad API into the existing action-binding layer. The
runtime already speaks actions, not key codes (`InputState` maps
KeyboardEvent.code → action), so gamepad buttons/axes become additional
bindings on the same actions — `ctx.input.isDown('jump')` is unchanged and
headless playtests keep working, since they drive actions directly. Needs:
stick deadzone handling, axis-to-action thresholds (left stick → move
actions), binding config in project input settings, editor UI for it.
Platformer-tier games expect controllers; this is the cheapest half of
closing the VVVVVV → Celeste gap.

## 12. Screen effects & shaders

Two tiers, in order:
- **Camera/screen effects (data-driven, do first)**: shake, flash, fade,
  zoom punch — `ctx.camera.shake(...)` etc., deterministic (seeded, so
  playtests can assert them) and exposed like tweens. This is most of
  "game feel" for the target tier.
- **Custom shaders (research)**: Pixi filters / fragment shaders as assets.
  Needs a sandboxing + asset-format story before design; don't block tier
  one on it.

## 13. Editor undo

Command-level undo/redo in the editor. Snapshots and diffs already exist in
the command registry; undo should be a registry concern (inverse ops or
snapshot stack), not a UI hack, so agents get `undo` over CLI/MCP too.
Pending since v0.1.

## 14. Content scale & performance

- **Perf ceiling**: stress-test and publish real numbers — hundreds vs
  thousands of active physical entities (Vampire-Survivors-like workloads).
  Add a physics broadphase and/or pooling if the numbers demand it; a
  benched example project keeps it honest.
- **Many-room games**: scene duplication/templating and tilemap-editing
  ergonomics so a 100+ room game is authorable without pain. Editor undo
  (§13) is a prerequisite for comfortable large-scale editing.

## 15. Editor UX friendliness (Jake, 2026-07-07, post-v0.8.0)

Make the editor approachable for the average user, not just agent-literate
developers:

- **Plain-language chrome**: visible Undo/Redo buttons in the toolbar.
  Engine-jargon like "diff" and "snapshot" is confusing to a normal user —
  rename/reframe those surfaces in user terms (e.g. "Review changes",
  "Checkpoint") or tuck them behind the agent-workflow panels where the
  audience expects them. Agent-facing CLI/MCP names stay as they are; this
  is about the human-facing editor copy.
- **Keybinds for common controls**: undo/redo already have Cmd+Z (verify
  redo binding); add the rest of the expected set — duplicate, delete,
  save/checkpoint, focus/zoom selection, play/stop — with a discoverable
  shortcut reference (menu hints or a cheat-sheet overlay).
- **Direct-manipulation transforms**: resize/reshape objects by dragging
  handles — selection gets the standard node handles on corners/edges
  (scale), plus rotation affordance; colliders/shapes reshape the same way
  (the polygon vertex editor exists — extend the pattern to
  width/height/radius on sprites, colliders, UI elements). Today moving is
  drag-native but sizing means typing numbers in the Inspector.

## Suggested wave grouping for the next run

- **Wave 0 (foundation, sequence first)**: scripting v2 — the Lua VM
  binding (wasmoon), the shared ctx surface across both languages, and the
  scene-management API touch the runtime, player, and core at once, and
  most other waves build on the new ctx surface. Include the no-chrome
  export rework here (kill the branded start screen, buildSettings-driven
  loading, silent audio unlock).
- **Wave A (runtime-heavy)**: rendering v2 (lighting, lines, debug,
  particles) + sprite animation — one agent owning packages/runtime.
- **Wave B (core+script API)**: math/events stdlib, pathfinding, physics v2
  properties + validation — parallel with A only if file ownership is split
  (stdlib/pathfinding are new modules; physics touches the same files as A's
  debug draw, so sequence those).
- **Wave C**: asset pipeline v2 + editor UX + docs/examples.
- **Wave D (flagship)**: embedded agent panel — after Wave 0 (agents need
  the complete scripting stdlib to build full games) and ideally after A/B
  so "make me this game" has the whole toolbox. Big enough to be its own
  run; the trust UX (command timeline, approve/revert) is where the design
  skills matter most. The PTY-terminal v1, though, is small enough to pull
  forward into any earlier wave — it needs nothing from Waves 0–C and
  immediately fixes the setup-friction disconnect.
- **Unassigned (added 2026-07-04, post-v0.6.0)**: UI widgets v2 (§7),
  gamepad input (§11), screen effects (§12), editor undo (§13), content
  scale & perf (§14). §7 + §11 together are the highest-leverage pair for
  raising the ceiling of what users and agents can ship (menus, HUDs,
  controller-native games); §12's camera-effects tier is small and pairs
  naturally with either.

## Adjustment: agent runs end-to-end by default (added 2026-07-09, Jake)

Revises the agent-panel "Trust UX" stance above. The default posture should be
**let the agent run**, not stop it for approval on every command. That
per-command gate is exactly the Claude Code "approve every tool call" friction
that most people bypass anyway, and an agent shouldn't be interrupted every few
seconds — it should execute a whole build end to end as needed.

- Default the embedded terminal / any `.mcp.json` and MCP permission mode we
  ship to a run-as-needed, auto-approve posture (agent executes through the
  full loop without per-step confirmation, like Claude Code's
  `--dangerously-skip-permissions` / accept-edits mode).
- Keep snapshot + structural diff + one-click revert as the **after-the-fact**
  safety net, not a per-command interruption. Review what it did once it's
  done, not before each step.
- Permission modes (read-only … build) stay available for anyone who wants
  them, but the default is "run end to end." Don't market the gating as the
  headline feature — the capability (agent drives the whole engine) is.

## The Road to 1.0 (added 2026-07-11, Jake) — THIS SUPERSEDES ALL PRIOR WAVE GROUPINGS

**The end goal, so every wave aims at it.** Definition of "fully fledged"
(1.0): a solo dev or an agent can take a 2D game from empty project to a
polished, distributable game without leaving the tool or hitting a wall.
Anti-bloat rule is binding: no feature lands unless it serves that loop;
"does too much" is a defect, not ambition.

Four waves remain. Each completes one loop rather than scattering features:

### Wave H — Iteration loop (v0.11)
Write→play→tweak becomes professional. Script hot-reload during play; live
property patching from the Inspector while playing (kills the Stop/Play
dance, incl. the old ambientLight live-edit gap); runtime error → click →
exact line in the Code panel; format-on-save (Lua + JS); code tabs
(multi-file editing); hover docs on ctx API; find/replace across scripts.
Every item doubles for agents (hot-reload = cheaper iteration; error→line =
faster self-fixes).

### Wave I — Content ceiling (v0.12)
What games can be stops being tooling-limited. Animation state machines
(SpriteAnimator is frame-loop only today); tilemap autotiling; particle
preview; live-linked prefabs / per-field overrides (the v0.9 deferral);
bulk asset import; prefab showcase example (the Wave F skip).

### Wave J — Ship your game (v0.13)
Exports stop being web-only. Desktop game export (per-platform player
packaging); signed/notarized engine builds; itch.io-ready output; project
templates for the empty-project moment.

### Editor ergonomics track — direct-manipulation editors (added 2026-07-11, Jake)
Certain editors must become intuitive surfaces, not lists of textfields and
buttons — think modern animation editors. Candidates, sized as wave items to
slot into v0.12.x/v0.13+ as capacity allows (each keeps the underlying
asset/command model unchanged — these are presentation layers over the same
agent-editable data):
- **Animator graph canvas**: states as draggable nodes, transitions as
  edges with inline condition badges, live highlight of the active state
  during play; the v0.12 list editor remains as the accessible/precise
  fallback view.
- **Animation timeline**: frame strip with scrubbing, per-frame duration
  handles, live preview of the selected clip.
- **Particle curve/gradient editors**: size/color-over-lifetime as draggable
  curves and gradient stops instead of paired number fields, with the live
  Scene preview reacting as you drag.
- **Autotile template painter**: visual 47-shape sheet mapper (click a
  shape, click a frame) instead of dropdown-per-shape.
This is distinct from the CUT visual logic editor (§10): no logic authoring,
only direct manipulation of existing data models.

### Wave K — 1.0 hardening (v1.0)
No features. Project-format stability guarantee (migrations + tests), docs
completeness pass, perf regression fences, onboarding polish, bug-tail
burn-down (the accumulated ledger minors), then tag 1.0.

### Explicitly CUT or post-1.0 (bloat guardrails)
- **Visual logic editor (§10): cut indefinitely.** Agents write code;
  humans get a real code editor and Lua. A node graph is a second, worse
  programming language to maintain. (Moves from "research" to non-goal.)
- **Custom GLSL shader assets: post-1.0 at most** — curated postEffects
  cover the 90% case; user shaders need a sandbox story.
- **Agent panel v2 chat UI, Codex auto-wiring, MCP resources, TS scripts,
  multi-instance components, plugin SDK, multiplayer research**: parallel
  track / post-1.0. None block fledgedness.
- Standing non-goals unchanged (no 3D, no built-in LLM calls, no cloud
  storage).
