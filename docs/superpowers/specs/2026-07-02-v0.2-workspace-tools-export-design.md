# Hearth v0.2 — Workspace, New Tools, and Web Export

Design spec and integration contract for the v0.2 milestone. Four tracks:
dockable editor workspace, game UI components, polygon colliders, audio —
plus production web export. Everything stays agent-native: every new
capability is a command/schema first, editor UI second.

## 1. Dockable workspace (editor)

Replace the fixed three-column shell with a docking workspace built on
**dockview** (`dockview` npm package, MIT — React docking layout with drag
tabs, splits, resize, serializable layouts).

- Panels registered by id: `hierarchy`, `scene`, `game`, `inspector`,
  `assets`, `console`, `diff`, `agent`. Scene and Game become sibling tabs
  in the center group by default (replaces the old center tab strip).
- Default layout mirrors today's: left Hierarchy, center Scene/Game tabs,
  right Inspector, bottom Assets/Console/Diff/Agent tab group.
- Layout serialized via dockview's `toJSON()` to
  `localStorage["hearth.layout.<projectId>"]`; restored on open; invalid or
  version-mismatched layouts fall back to default.
- Toolbar gains a **View** menu: checkbox per panel (open/close), and
  "Reset layout". Closing the last panel of a group collapses the group
  (dockview handles this). Console badge behavior is preserved by keeping
  the unread count in the store and rendering it in the panel tab title via
  dockview's tab component API.
- Theme dockview with Hearth's existing CSS variables (dark editor theme,
  ember accent). No default dockview theme visible anywhere: custom tab
  chrome, separators, drop-target overlay tinted with the ember accent.
- Keyboard: no new global shortcuts in v0.2.

Store changes: `centerTab`/`bottomTab` are removed; `consoleUnread` stays.
DiffPanel refresh-on-focus moves to a dockview `onDidActivePanelChange`
listener.

## 2. Game UI components (core + runtime + editor)

New component `UIElement` (schema lives in `packages/core/src/schema/components.ts`):

```ts
UIElement: {
  anchor: 'top-left'|'top'|'top-right'|'left'|'center'|'right'|
          'bottom-left'|'bottom'|'bottom-right'   // default 'top-left'
  offset: Vec2                                    // px from anchor, default {0,0}
  interactive: boolean                            // default false
}
```

- An entity with `UIElement` renders in **screen space**: camera position
  and zoom do not affect it. Its visuals come from the same `Text` /
  `SpriteRenderer` components every other entity uses. `Transform.scale`
  and `rotation` still apply; `Transform.position` is ignored (anchor +
  offset position it).
- UI renders above all world layers; within UI, existing `layer` fields
  order elements.
- `interactive: true` enables pointer hit-testing against the element's
  rect (SpriteRenderer width/height or measured Text bounds). Events call a
  new script hook on the same entity:
  `onUiEvent(ctx, event)` with `event = { type: 'click'|'press'|'release'|'enter'|'exit' }`.
- Headless runtime: UI elements exist and are inspectable;
  `runtime.sendPointer(x, y, kind)` lets playtests simulate clicks
  (screen coordinates). No rendering needed.
- Pixi host: UI layer is a separate container overlaying the world,
  re-anchored on resize.

## 3. Polygon colliders (core + runtime + editor)

Extend `Collider`:

```ts
shape: 'box'|'circle'|'polygon'
points: Vec2[]   // local space, used when shape==='polygon'; default is a triangle
```

- **Convex only in v0.2.** `validateProject` errors on `polygon` colliders
  with fewer than 3 points, duplicate consecutive points, or non-convex
  winding, with a message telling the user/agent to split concave shapes
  into multiple entities.
- Runtime physics (`packages/runtime/src/physics.ts`): SAT for
  polygon–polygon, polygon–box, polygon–circle, with MTV resolution for
  dynamic bodies and overlap-only reporting for triggers. Points are
  transformed by entity scale and rotation. Existing box/circle behavior
  must not change (current tests keep passing).
- Editor: with a polygon-collider entity selected, the Scene view offers
  **Edit Points**: drag vertex handles, click a midpoint handle on an edge
  to insert a vertex, alt/right-click a vertex to delete (min 3 enforced).
  Every mutation goes through the existing `setComponentProperty` command
  (`Collider.points`), so diff/snapshot/revert see it.

## 4. Audio (core + runtime + editor)

- Script API: `ctx.audio.play(assetRef, { volume?, loop? }) -> handleId`,
  `ctx.audio.stop(handleId | assetRef)`. `assetRef` is an asset id or name.
- `AudioSource` component becomes functional: `autoplay` plays on scene
  start with its `loop`/`volume`; scripts can also `ctx.audio.play` any
  audio asset directly.
- Headless runtime: no sound, but every play/stop is recorded in the run
  report as `audioEvents: [{ frame, assetId, action: 'play'|'stop' }]` so
  agents and playtests can assert audio behavior.
- Pixi host: Web Audio implementation — fetch + decodeAudioData, gain node
  per playback, master gain, all playback stopped on runtime stop. Browser
  autoplay policy: audio context resumes on first user gesture (the web
  export's click-to-start screen guarantees a gesture).
- **Procedural sounds** (`packages/core/src/assets/sounds.ts`): a
  deterministic synthesizer (seeded PRNG, no Date/Math.random) that writes
  real 16-bit PCM WAV files. Presets: `coin`, `jump`, `hit`, `laser`,
  `powerup`, `explosion`, `blip`. New command `createSound { name, preset,
  seed? }` (permission `asset-edit`) writes `assets/sounds/<slug>.wav` and
  registers an `audio` asset. CLI: `hearth create sound <name> --preset
  coin`. FsLike gains binary file support (`writeFile` accepting
  Uint8Array) if it doesn't already have it.
- Editor: Assets panel shows audio assets with a play/preview button;
  Inspector's AudioSource gets an asset picker; "experimental" labels for
  audio are removed.

## 5. Production web export

`hearth export web` produces a static, self-contained playable build —
the moral equivalent of Godot's HTML5 export.

- **Player bundle**: `packages/runtime` gains a build step
  (`scripts/build-player.mjs`, esbuild) producing
  `packages/runtime/player/hearth-player.js` — an IIFE bundling the runtime
  + Pixi host that exposes `window.HearthPlayer.boot({ mount, bundle })`.
  `bundle` is the parsed project bundle (below). The player renders a
  click-to-start screen (satisfies audio autoplay policy), then runs the
  initial scene, scaled to fit the window (letterboxed, integer-friendly).
- **Project bundle** (`project.bundle.json`): `{ project, scenes: [...],
  scripts: { [path]: source }, assets: [{ id, name, type, path | dataUri }] }`.
- **Command** `exportWeb { outDir = 'export/web', singleFile = false }`
  (permission `build`): validates first (like buildProject), then writes
  `index.html`, `hearth-player.js`, `project.bundle.json`, and `assets/`.
  With `singleFile: true` it emits one `index.html` with the player inlined
  and assets as data URIs. The player JS is provided by the host through a
  new optional `CommandContext.resources.getPlayerBundle()` — wired in the
  CLI, MCP server, and editor project server from
  `HEARTH_TOOLS_DIR/hearth-player.js` (packaged app) or the runtime
  package's `player/` dir (repo checkout); tests inject a stub. If the
  player bundle can't be found, the command fails with a clear
  `MISSING_RESOURCE` error naming the expected locations.
- **index.html template** (embedded string in core): game title, themed
  loading/start screen, fullscreen button, canvas letterbox scaling,
  no external requests.
- CLI: `hearth export web [--out DIR] [--single-file] [--zip]`; `--zip`
  additionally writes `<name>-web.zip` (store-only minimal zip writer in
  the CLI package — node-only) ready for itch.io upload.
- MCP: `export_web` tool. Editor: an **Export** dialog on the toolbar
  (target: folder / single file; destination picker in the desktop app;
  reveal-in-folder button on success) calling a new `/api` export route.
- `buildProject` stays (portable project folder) with its description
  updated to point game exports at `exportWeb`.

## Testing

- Core: schema/validation tests (polygon convexity, UIElement defaults),
  createSound determinism (same seed → identical bytes), exportWeb output
  shape (folder + single-file) against MemoryFileSystem with a stub player.
- Runtime: SAT collision suite (poly-poly, poly-box, poly-circle, triggers,
  rotation/scale), UI hit-testing + onUiEvent dispatch via sendPointer,
  audioEvents recording, AudioSource autoplay.
- Playtests: example playtests keep passing; at least one example gains a
  UI + audio + polygon showcase and a playtest asserting an audio event.
- Editor: typecheck + HEARTH_SMOKE packaged smoke; layout persistence
  covered by a small unit test on the (de)serialization helper.
- CI: player bundle build added to `build:packages` so release artifacts
  and the packaged app ship it (`build-electron.mjs` / `assemble-app.mjs`
  copy it next to hearth-cli.mjs).

## Non-goals (v0.2)

Native executable export (roadmap: per-platform player templates), concave
polygon decomposition, UI layout containers/flex, audio mixing buses,
floating multi-window panels beyond what dockview provides.
